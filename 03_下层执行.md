# 03 下层执行：计划、批任务拆解与主从编排

## 核心定义

下层执行，是把上层规则转化为一次次可控的任务执行。

它关注的不是“AI 能不能做”，而是：

- 任务如何进入？
- 怎么判断范围？
- 如何拆批？
- 哪些先串行，哪些能并行？
- 子 agent 的边界在哪里？
- 主 agent 如何集成和验证？
- 做到哪里必须停下来审查？

下层执行的目标是：

> 把开放式任务降维成可计划、可验证、可回滚的小闭环。

## 1. 任务进入：先识别任务类型

不同任务进入系统后，走不同执行策略。

| 任务类型 | 特征 | 推荐处理 |
|---|---|---|
| 小修小补 | 范围明确、低风险 | 直接执行 + 简短验证 |
| 复杂功能 | 多模块、多状态、多依赖 | 先计划、再 plan-review、再执行 |
| 架构/契约变更 | 影响上层规则或跨模块协议 | 单独成批，必须审查 |
| 批量迁移 | 多页面、多端、多文件重复工作 | 建 manifest、分批、统一 recipe |
| 自动修 bug | 外部任务输入、需要诊断和人类确认 | 队列 + 诊断 + 人工卡点 |
| 长任务续作 | 会话可能中断或跨时间执行 | handoff 文档 + cron/watch/loop |
| 无人值守 | AI 自主跑一段时间 | 需要 oracle、门禁、审核队列 |

不要让所有任务走同一流程。

## 2. 计划：把任务变成可控变更单元

复杂任务必须先写计划。

计划不是为了形式，而是为了让 AI 明确：

- 做什么
- 不做什么
- 听谁的
- 怎么拆
- 怎么验收
- 什么时候停

### 2.1 计划必须包含的内容

```md
## 目标
本次要达成什么结果。

## 非目标
本次明确不做什么。

## 真相源
架构、业务、视觉、契约、运行事实分别来自哪里。

## 范围
会影响哪些模块、页面、接口、文档、测试。

## 执行步骤
按依赖顺序拆解。

## 风险
最可能错在哪里。

## 验收
怎么证明完成。

## 停止条件
什么情况下必须停下来审查或回问。
```

### 2.2 计划中的非目标很重要

AI 容易顺手扩张。

非目标的作用是告诉 AI：

- 看到相关问题也不要顺手修。
- 发现新需求先登记，不直接做。
- 不要把本批变成“顺便重构”。
- 不要把临时简化伪装成完成。

## 3. 批任务拆解

批任务的核心不是“平均分工”，而是按风险、依赖和验证方式拆。

### 3.1 不要按时间拆

错误拆法：

```text
今天做一半页面，明天做另一半。
```

更好的拆法：

```text
先做共享基建
再做彼此独立的页面
最后统一集成验证
```

AI 执行不适合用时间作为主要边界，因为时间不可控。

更好的边界是：

- 文件范围
- 模块边界
- 共享依赖
- 风险等级
- 验收方式
- 回滚粒度

### 3.2 常见拆解维度

| 维度 | 拆法 |
|---|---|
| 共享依赖 | 先做基建、组件、契约、mock，再扇出业务 |
| 风险等级 | 高风险写操作单独成批 |
| 验证方式 | 需要真机/真账号/截图的集中验证 |
| 业务边界 | 一页、一域、一能力、一状态机 |
| 变更性质 | 基建、功能、修复、测试、文档分开 |
| 回滚粒度 | 能独立回滚的一组作为一批 |

### 3.3 批次大小原则

一个批次应该满足：

- 一眼能审
- 可独立验证
- 可独立回滚
- 目标单一
- 不跨太多边界

如果一批同时包含：

- 新架构
- 新页面
- 新接口
- 新样式系统
- 新测试框架
- 文档迁移

那就过大了。

## 4. Phase 编排：Phase 0 / Phase 1 / Phase 2

批任务中最稳定的编排模式是三段式。

```text
Phase 0：共享基建串行先行
Phase 1：独立任务并行扇出
Phase 2：主 agent 集成验证
```

### 4.1 Phase 0：共享基建串行先行

Phase 0 做所有会被多个子任务依赖的东西。

包括：

- 目录结构
- Query / API / Repository 基座
- key 工厂
- mock / fixture
- i18n 空块
- 共享 UI 类或组件
- 公共类型
- 测试工具
- lint / grep / dep rules

原则：

> 共享件没就绪，不允许扇出并行。

否则子 agent 会各自补一套共享件，造成冲突和分叉。

### 4.2 Phase 1：独立任务并行扇出

Phase 1 可以并行，但必须限制边界。

典型规则：

- 一个 agent 只负责一个页面、一个模块或一个能力。
- 只能改自己的文件。
- 不得改共享件。
- 需要共享件改动时，停下报告主 agent。
- 每个 agent 做逻辑级自测。
- 不要求每个 agent 做最终像素/真机/全量验证。

这样并行才安全。

### 4.3 Phase 2：主 agent 集成验证

Phase 2 必须由主 agent 串行完成。

原因：

- 子 agent 看不到完整上下文。
- 子 agent 容易各自为政。
- 视觉、真机、真账号、console、端到端验证必须统一口径。
- 集成问题往往跨文件，不能分散处理。

Phase 2 做：

- 全量静态检查
- 构建
- grep 闸门
- 浏览器/真机验证
- 截图对账
- console 检查
- 文档同步
- change-review
- 修复集成问题

即使基建阶段的单元测试已经全绿，也不能因此跳过手动/端到端的 smoke 验证——多个 phase 组合出的集成层 bug，往往只有在真实调用方出现后才会暴露，单测构造的对象常常没有真实下游消费者。

### 4.4 Phase 纪律

phase 不是时间盒，而是可审查、可回滚的变更单元。

推荐纪律：

- 单个 phase 控制文件数量，避免一批大到无法审查。
- 单端任务不跨端混改。
- 单类型任务不混做，例如基建、配置、测试、feature、文档分开。
- 不跨 phase 合并执行。
- 每个 phase 结束后做 change-review。
- 高风险 phase 结束后再 commit / tag / 归档审查记录。
- 终止条件不基于“时间到了”，而基于 scope 失控、返工率高、真相源不清或验证无法完成。

phase 的目标不是让 AI 忙满一段时间。

phase 的目标是让人类能看懂、能审完、能回滚。

### 4.5 Scope 失控的信号

出现这些情况，应停止当前 phase，重新拆解：

- 文件数量显著超过计划。
- 任务同时触碰 UI、接口、架构、测试、文档多类边界。
- 为了完成当前任务，不断新增共享抽象。
- 子 agent 需要改共享件才能继续。
- 验收方式从简单自测升级成真机、真账号、多状态矩阵。
- 改动开始依赖未确认业务规则。
- 发现旧文档和当前事实冲突。

停止不是失败。

停止是把失控变更重新变成可治理变更。

## 5. 子 agent 与主 agent 分工

### 5.1 子 agent 适合做什么

子 agent 适合做：

- 独立页面
- 独立模块
- 单一域 DTO 移植
- 单一审查维度
- 单一文档章节
- 重复但边界清晰的工作

### 5.2 子 agent 不适合做什么

子 agent 不适合做：

- 改共享基建
- 改全局架构
- 做最终集成判断
- 做跨任务裁决
- 处理用户临时决策
- 做高风险 git 操作

### 5.3 主 agent 的责任

主 agent 不是简单调度器，而是集成负责人。

必须负责：

- 定义 Phase 0 共享件
- 给子 agent 明确边界
- 检查子 agent 是否空跑
- 抽查关键文件
- 统一集成验证
- 汇总文档与技术债
- 发起 change-review
- 处理 must-fix

经验教训：

> 子 agent 偶发空跑或误解边界，主 agent 必须核对交付，不能只看报告。

### 5.4 子 agent 模型经济学

分完工，还要分**模型档位**。一刀切（全用大模型 = 烧钱；全用小模型 = 烧质量）都是错的。按角色三档：

| 档位 | 角色 | 模型 |
|---|---|---|
| 检索 / 广撒网 | 找调用方、扫同类实现、拉资料 | 小模型 |
| 常规判断 | 规范核对、清单式预检 | 小模型可胜任（有清单兜底） |
| 核心判断 / 对抗复核 | 正确性主判断、blocking finding 证伪 | **主力模型，不降级** |

判断标准一句话：

> 输出**直接决定 blocking verdict** 的 agent 不降级；输出还需主 agent 消化核实的可降级。

一个反直觉的隐性成本：给弱模型审查加的防误报补丁（"必须引用 file:line""主 agent 先自行核实再采纳"）本质是**主 agent 在为弱模型买单**——弱模型省下的钱，以主 agent 复核成本的形式还回来。总则：**省钱在检索，保质在核查**。

### 5.5 并行写操作的物理隔离：worktree

子 agent 并行执行时，如果任务拆分不能完全保证各自改动区域零重叠（比如都可能触及同一个公共文件），仅靠"任务边界约定"不够可靠——应该用 git worktree 给每个并行写入的 agent 一份物理隔离的工作副本，而不是依赖事后合并处理冲突。这不限于无人值守长任务：常规的多 agent 并行开发同样适用，一旦区域重叠风险存在，隔离应该是默认选项，而不是出问题后才补救的手段。

### 5.6 并行数量上限：谁来定

子 agent 并行数不是越多越好——上限该由**主 agent 审得完**决定，而不是工具允许的并发上限。

工具层面的并发数字是物理硬顶，不是治理目标。例如某些 orchestration 工具会把并发限制在"核数相关的一个值"、总量限制在一个很大的数——这保证的是"不把机器打爆"，不代表"开到这个数字是安全的"。物理硬顶防止系统崩，治理上限防止审查退化成只看子 agent 的自我报告、不核实交付内容（案例库 #6 子 agent 空跑）。

如果全局或项目规则没有另行规定数量，给一个可以直接套用的默认值：

> **单批有写权限、可能触碰共享资源的子 agent，并行数默认不超过 4 个。**

理由：§5.3 要求主 agent 对子 agent 的交付"抽查关键文件"逐个核对，超过这个量，人（或主 agent 自己）会不自觉地退化成只扫一眼报告结论，而不是真的复核代码。

这个默认值有一类例外可以放宽：**纯只读检索类子 agent**（Explore、扫同类实现、拉资料，不写任何文件）互相之间没有冲突风险，交付也不需要逐个核对"是否正确合并"——上限可以直接跟着"任务能拆出多少个独立检索单元"走，不必卡在 4。

可以直接套用的判断公式：

```text
本批并行数 = min(
  工具的并发硬顶,
  任务能拆出的独立单元数,
  主 agent 愿意/能够逐个核对交付的数量   ← 有写权限时通常是瓶颈项，默认 4
)
```

### 5.7 开子线前必须先检查：隐性共享文件

§4.1 的"共享基建先行"最容易在执行时打折扣的地方，不是大家都认得出的公共组件、公共类型，而是那些看起来可以"顺手加一行"的**单文件共享资源**——典型例子是路由表。

用路由表展开成一条可执行判断：

- 如果本批有多个页面级子 agent，且每个子 agent 完成页面后都要往同一份路由表（`routes.ts`、`App.tsx` 里的 `<Route>` 列表、导航注册表）里加一条——这份路由表在 Phase 1 开始前就已经是共享冲突点，不能留到"大家写完了再合"。
- 处理方式二选一，必须在 Phase 0 定下来，不能拖到 Phase 1 才发现：
  - **(a) 骨架化占位**：Phase 0 先把每个子 agent 未来要挂载的路由项占位建好（占位组件 + 固定路径），子 agent 只需要"替换"自己的占位块，不需要"插入"到共享文件的任意位置——插入位置不确定是合并冲突的主要来源，替换固定占位块没有这个问题。
  - **(b) 改动权收归主 agent**：子 agent 完成页面后只报告"需要挂载到 `/xxx` 路径"，路由表本身只由主 agent 在 Phase 2 统一追加。

路由表只是一个例子，同类"隐性共享单文件"还有：

| 类型 | 例子 |
|---|---|
| 注册表 | 路由表、DI 容器注册、Redux/Store slice 注册 |
| 索引 / 导出清单 | barrel file（`index.ts` 统一导出）、组件清单 |
| 键值表 | i18n key 表、埋点事件表、feature flag 表 |
| 目录索引 | 文档 README 目录、测试 fixture 索引 |

判断标准就一句：

> 如果两个子 agent 各自完成任务后，最终都会往**同一个文件**里追加内容，这个文件在 Phase 1 开始前就不能是"从零开始"的状态。

这和 §5.5 的 worktree 不是二选一，是两层不同问题：worktree 解决"物理上不冲突"（各自独立副本，最后合并）；本节解决"逻辑上不冲突"（从设计上让共享文件根本不需要被多个 agent 同时改）。即使上了 worktree，仍然建议先做骨架化——减少的是最后合并时要人工介入的冲突量，两者可以叠加使用。

### 5.8 模型自动选择该定义在哪层 CLAUDE.md

§5.4 回答"什么任务该配什么档位"，这里回答治理层面的问题：这条规则该写在哪份文件里，才能既全局生效，又不至于换个项目、换个模型供应商就要大改。

按 [`01_上层定法.md`](./01_上层定法.md) §4.5 的规则分层拆成两层：

- **三档划分原则**（检索/广撒网、常规判断、核心判断与对抗复核的边界，以及"输出是否直接决定 blocking verdict"这条判断标准）——这是"AI 如何协作"的全局工作规则，跟具体项目、具体模型供应商无关，应该写进**全局 CLAUDE.md**。
- **档位 → 具体模型名的映射**——这本质是项目/团队的成本决策（用哪家供应商、预算多紧），理论上属于**项目 CLAUDE.md** 的管辖范围，不该硬编码进全局规则。

一个具体示范（作者本人当前的全局 CLAUDE.md 原文）：

```md
## 子 Agent 模型（按角色三档）
- 检索/广撒网（Explore、找调用方、扫同类实现、拉资料）→ sonnet
- 常规判断（规范核对、清单式预检）→ sonnet
- 核心判断与对抗复核（主判断、blocking finding 证伪）→ 不指定 model，继承主模型
- 判断标准：输出直接决定 blocking verdict 的不降级；输出还需主 agent 消化核实的可降级
```

严格按上面的分层，这份规则其实把"原则"和"当前映射"写在了同一层（全局）。这不是错误，而是可以接受的现状：**分层本身有维护成本，只有在真正出现"同一原则、不同项目要不同映射"的需求时，才值得把"档位→模型"那一行下沉到项目 CLAUDE.md，让全局只留原则**。只有一套映射、还没有第二个项目提出不同要求之前，强行拆两层文件只是增加找规则要跳转的成本，没有换来实际收益。

拆分的触发信号：出现第二个项目需要不同的档位映射（比如某项目成本敏感、要把检索档降到更便宜的模型；或换了模型供应商，档位名不再适用）——这时才把映射行搬进项目 CLAUDE.md，全局 CLAUDE.md 改回只留原则那几行。

落地形态还可以再往下一层机制化（呼应 [`07_工具链治理.md`](./07_工具链治理.md) §6"文档定法→机制执法"）：如果编排工具支持按"角色名"而非"具体模型名"传参（例如按角色传 `role: '核心判断'` 而不是硬编码具体模型名），角色到模型的映射就能集中改一处生效，不必每次开 agent 时人工判断该用哪个模型名。

### 5.9 什么时候该并发

前面几节都假设"已经决定要并行"，这里退一步：并发不是默认选项，不是"能拆就该拆"。

值得开并发，需要三个条件同时成立：

- **无决策依赖**：子任务之间不需要先看到彼此的结果才能定范围。
- **可独立验证**：呼应 §3.3 批次大小原则。
- **协调成本 < 省下的时间**：这条最容易被忽略。开并发有真实开销——定义边界、写清楚每个子任务的输入输出、逐份审查交付（§5.3 / §5.6）。任务数量少、或单任务本身很快时，这笔开销可能倒挂，串行反而更快。

给一个可以直接套用的锚点：

> **可拆分的独立单元数 ≤ 2~3 个时，默认直接串行做，不值得为了"看起来更并行"而拆分。**

不该并行的信号：

- 任务之间存在顺序依赖，后一个需要前一个的产出才能定范围
- 共享资源无法骨架化（§5.7 处理不了，只能靠主 agent 串行改）
- 验证方式要求整体一起看——比如某类一致性检查，拆开单独看反而看不出问题
- 任务过于琐碎，写清楚边界的成本比直接动手做还高

一句话判断标准：

> 并发的收益来自"任务数量 × 单任务耗时"的节省，成本来自协调和审查；只有前者显著大于后者时才值得开。

### 5.10 自动并发设计：默认并行还是默认加同步点

§5.9 回答"要不要并行"，这里回答更进一步的问题：怎么把这个判断变成默认机制，而不是每批任务都重新想一遍。

一个常见误区：把"分阶段"等同于"每阶段都要等全体到齐才能进下一阶段"。这个默认同步点（下称"屏障"）的代价是，最慢的一个任务会拖慢所有任务进入下一步的时间——而多数情况下，下一步根本不需要看到"全体"的结果，只需要看到"自己"上一步的结果。

更省时间的默认设计是反过来的：**默认让每个独立单元自己流过各阶段，互不等待**（一个子任务已经进入第二阶段，另一个还在第一阶段，各走各的）。只有满足下面任一条件时，才显式加一个屏障：

| 触发条件 | 例子 |
|---|---|
| 下一步要对全体结果去重 / 合并 | 汇总多个来源的发现后统一处理，避免同一个问题被处理两次 |
| 下一步要根据"总数"做判断 | 这批一个问题都没发现，直接跳过后续所有环节 |
| 下一步依赖"和其他结果的比较" | 从多个方案里挑出最优的一个 |

除了这三类，其余场景都应该默认走"各自流转、互不等待"，不要因为"看起来更整齐"就习惯性同步。

这不是抽象设想——这是我自己在编排机制上采用的默认：不显式声明同步点时，各条目独立流转，互不等待；只有明确要求"等所有人都做完"才是屏障。这个默认帮助避免了"图省事习惯性同步"这种隐性拖慢。

和 §5.6（并行数量上限）的关系：两个旋钮是正交的，不要混为一谈。数量上限决定"同时有多少个单元在跑"；屏障决定的是"阶段之间要不要等全体到齐"。限制了数量不代表就不需要考虑屏障，两件事要分开判断。

## 6. 逐页 / 逐模块 Recipe

对重复任务，应沉淀 recipe。

一个典型 recipe：

1. 读输入源：架构、业务、视觉、契约、现有实现。
2. 做范围裁定：哪些做，哪些不做，哪些登记债务。
3. 接数据：只经规定层，不绕过统一入口。
4. 渲染或实现：遵守样式、埋点、状态、错误处理规则。
5. 自测：本模块逻辑路径跑通。
6. 回填文档：状态、债务、决策、跳转关系。
7. 交给主 agent 集成验证。

Recipe 的价值是：

- 降低每次重新商量的成本
- 防止 AI 漏步骤
- 让新 agent 可以接手
- 让批任务可并行

迁移 / 重写场景下 Recipe 的输入源怎么定（多源分维度授权）、"一致"怎么证明（parity 对账），见 [`09_迁移与重写治理.md`](./09_迁移与重写治理.md)。

## 7. 状态分类与接线

批量还原或页面构建中，状态必须先分类。

一种通用分类：

| 类型 | 含义 | 处理方式 |
|---|---|---|
| 持久业务状态 | 页面长期展示的状态，如空态、审核中、失败 | 可进入状态切换器或测试矩阵 |
| 触发弹窗 | 点击按钮出现的瞬时态 | 通过真实按钮触发，不作为主状态 |
| 流程步骤 | 表单步骤、确认步骤、推进步骤 | 通过 Next / Confirm 到达 |
| 共享浮层 | 多页复用的 sheet/dialog/card | 抽共享件，统一接线 |

先分类，再实现。

否则容易：

- 把弹窗做成伪页面状态
- 把真实流程藏进调试切换器
- 重复实现共享浮层
- 漏掉状态回填

## 8. 验收设计

验收必须分层。

| 层级 | 谁做 | 验收内容 |
|---|---|---|
| 子任务自测 | 子 agent | 自己范围的逻辑、基本渲染、局部测试 |
| 集成验证 | 主 agent | 全量构建、跨模块、视觉、console、真机 |
| 对抗审查 | review agent / 主 agent | 范围、架构、正确性、验证充分性 |
| 人工卡点 | 用户/负责人 | 产品判断、风险接受、合并/发版 |

验收不应该只靠一个环节。

## 9. 停止条件

执行中遇到以下情况应停止或升级：

- 真相源冲突无法按规则裁决
- 需要修改架构不变量
- 发现新需求超出本批范围
- 需要高风险写操作
- 关键验证环境不可用
- 视觉源缺失且无法合理降级
- 契约不明且影响真实请求
- 子 agent 需要改共享件
- 自动化验证失败且原因不明

停止不是失败。

对 AI 来说，知道何时停比盲目继续更重要。

## 10. 下层执行检查清单

开始执行前确认：

- 本任务类型是什么？
- 是否需要先写计划？
- 是否声明了非目标？
- 是否声明了真相源？
- 是否有明确验收？
- 是否拆成可审的小批次？
- Phase 0 共享件是否先行？
- Phase 1 并行边界是否清楚？
- Phase 2 集成验证由谁负责？
- 什么情况下必须停？

执行完成后确认：

- 计划内目标是否完成？
- 是否有越界改动？
- 是否产生新技术债？
- 是否需要更新 Spec？
- 是否需要更新 workflow 或 checklist？
- 是否已经进入 change-review？
