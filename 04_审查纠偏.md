# 04 审查纠偏

## 核心定义

审查纠偏是 AI 协作工程治理系统里的制动器。

它不是礼貌性复核，也不是“看一眼没问题”。它的职责是：

- 发现目标漂移
- 发现真相源误用
- 发现架构破坏
- 发现实现遗漏
- 发现验证造假
- 发现文档与代码脱节
- 在风险进入主干前叫停流程

一句话：

> 审查不是盖章，而是让 AI 的产出必须经得起反证。

## 1. 为什么 AI 协作必须有对抗性审查

AI 的常见问题不是“不工作”，而是“看起来完成了”。

典型表现：

- 计划写得完整，但遗漏关键依赖。
- 代码改了很多，但没有触达真正问题。
- 说测试通过，但测试没有断言。
- 说参考了文档，但文档里根本没有对应结论。
- UI 看起来接近，但状态、边距、交互不符合真相源。
- 子 agent 输出很多内容，但实际没有读取核心文件。
- 修复一个 bug，同时引入更隐蔽的架构债。

所以审查必须带有对抗性。

对抗性不是否定 AI，而是主动问：

- 这个结论从哪里来？
- 有没有相反证据？
- 是否越过了任务边界？
- 是否符合上层不变量？
- 是否有真实验证，而不是口头验证？
- 如果今天合入，明天谁会被影响？

## 2. 审查在治理系统中的位置

审查位于执行流的关键关口。

```text
目标输入
  → 计划拆解
  → Plan Review
  → 执行实现
  → 自验证
  → Change Review
  → 交付 / commit / 发布
  → 经验回流
```

Plan Review 审的是“准备怎么做”。

Change Review 审的是“实际做成了什么”。

两者不能互相替代。

计划通过，不代表实现一定正确。

实现看似可用，也不能倒推计划合理。

## 3. Plan Review：执行前审查

Plan Review 的目标是防止任务一开始就走偏。

它重点检查：

- 目标是否清楚
- 范围是否收敛
- 真相源是否明确
- 依赖是否识别
- 风险是否暴露
- 验收标准是否可验证
- 执行顺序是否合理
- 是否需要人工卡点

### 3.1 Plan Review 必查项

| 检查项 | 要问的问题 |
|---|---|
| 目标 | 这个计划要解决的用户问题是什么？ |
| 范围 | 哪些做，哪些明确不做？ |
| 真相源 | 业务、视觉、架构、接口分别以谁为准？ |
| 不变量 | 有没有触碰架构 constitution？ |
| 依赖 | 是否依赖外部账号、设备、环境、权限、数据？ |
| 拆解 | 是否可以分批执行和独立验收？ |
| 风险 | 最可能失败在哪里？ |
| 验收 | 如何证明完成，而不是只说完成？ |
| 卡点 | 哪些动作必须等人确认？ |
| 回滚 | 出错后如何停住或恢复？ |

### 3.2 Plan Review 的常见 verdict

Plan Review 应该给出明确结论，而不是模糊意见。

可用 verdict：

- `APPROVE`：计划完整，可以进入执行。
- `APPROVE WITH CHANGES`：总体方向可行，但 must-fix 项必须先修。
- `RETHINK`：计划基础假设错误，需要重写。

例子：

```text
Verdict: APPROVE WITH CHANGES

Must-fix:
- 补充真相源优先级：生产接口 > 已验证行为 > 旧文档。
- 将批量迁移拆为 Phase 0 基础设施、Phase 1 页面并行、Phase 2 集成验收。
- 增加人工卡点：commit 前必须输出变更范围和验证证据。
```

### 3.3 Plan Review 的问题分类

Plan Review 不只看“计划有没有写”。

推荐按这些类别审：

| 分类 | 关注点 |
|---|---|
| DESIGN | 方案是否过度设计、抽象层级是否正确、是否绕开成熟做法 |
| EDGE CASE | 网络、空态、并发、后台切换、迁移、兼容等真实边界 |
| INCOMPLETE | 目标、范围、文件、步骤、依赖、顺序是否缺失 |
| CONVENTION | 是否违反 AGENTS、CLAUDE、项目规范、架构边界 |
| RISK | API、类、路径、外部依赖是否真实存在，是否有隐藏副作用 |
| IMPACT | 是否影响公共 API、共享模型、序列化、跨模块调用 |
| MAINTENANCE | 是否引入隐式契约、魔法值、顺序依赖、长期理解成本 |

Plan Review 的价值在于提前打断错误方向。

计划阶段发现问题，修的是文字和顺序。

实现后才发现问题，修的可能是几十个文件。

## 4. Change Review：交付前审查

Change Review 的目标是防止错误产物进入下一阶段。

它关注的不是“有没有努力”，而是：

- 改动是否解决目标问题
- 是否引入新问题
- 是否符合架构和业务约束
- 是否有证据证明
- 是否留下未登记风险

### 4.1 Change Review 必查项

| 检查项 | 要问的问题 |
|---|---|
| 范围一致性 | 改动是否只覆盖本次任务范围？ |
| 真相源一致性 | 实现是否来自正确来源？ |
| 架构一致性 | 是否破坏模块边界、数据流、状态管理？ |
| 行为正确性 | 正常路径、异常路径、空状态是否正确？ |
| 验证证据 | 测试、截图、日志、真机、像素检查是否真实存在？ |
| 文档同步 | 是否更新受影响的 spec、README、技术债、handoff？ |
| 债务登记 | 遗留问题是否显式记录？ |
| 回归风险 | 旧功能、旧入口、旧数据是否受影响？ |

### 4.2 Change Review 的常见 verdict

可用 verdict：

- `APPROVE`：可以交付。
- `APPROVE WITH CHANGES`：存在 must-fix，修完后可交付。
- `REQUEST CHANGES`：存在阻断问题，不能交付。

例子：

```text
Verdict: REQUEST CHANGES

Blocking:
- 提交声称完成接口切换，但仍有两个调用点使用旧 path。
- 测试只覆盖 mock 成功路径，没有覆盖服务端返回空数组。
- README 中的启动命令已失效，文档未同步。
```

### 4.3 Change Review 的问题分类

Change Review 要证明“实际改动能交付”。

推荐按这些类别审：

| 分类 | 关注点 |
|---|---|
| PLAN GAP | 实现是否漏掉计划项，是否出现未解释偏离 |
| DESIGN | 抽象、耦合、状态管理、生命周期、静默失败 |
| ROBUSTNESS | 异步失败、空值、超时、重试、清理、崩溃风险 |
| PERFORMANCE | 重复计算、监听泄漏、N+1、主线程阻塞 |
| CONVENTION | 项目规则、命名、禁用模式、架构边界 |
| QUALITY | 明显逻辑错误、死代码、TODO、未使用变量、资源泄漏 |
| LINTER | lint/analyze/test 中由本次改动引入的问题 |
| RECURRING | 历史 review 中反复出现的问题 |

Change Review 必须读取实际变更文件和上下文。

只看 diff 容易错过：

- 调用方影响
- 文件级约定
- 生命周期关系
- 旧逻辑保护条件
- 文档同步缺口

### 4.4 Plan Review 的 must-fix 在计划阶段闭环

Plan Review 提出 must-fix 后，正确处理不是把问题登记成账本带进执行期，而是：

> 直接修订计划文档本身 → 增量 re-review → 直至 APPROVE，然后才进入执行。

理由：此刻还没开工，修改计划文字是**零成本的源头修复**。计划修好了，问题就不存在了——不需要任何跨阶段跟踪机制，修好的计划就是闭环。

因此 Change Review 不承担"关闭 Plan Review 遗留项"的职责：进入执行的计划必然已是 APPROVE 态；用户显式跳过 plan-review 直接执行的，也没有账要还。

> 历史注：本知识库早期版本设计过"plan-review 落盘 pending → change-review 逐项关闭（PR-RESOLVED / PR-UNRESOLVED）"的跨阶段闭环机制，后被整体废弃——它防的问题可以在更源头零成本消掉。这是一个典型的治理债样本，定义与判断标准见 [`06_技术债与经验回流.md`](./06_技术债与经验回流.md) §4.1「治理债」。

## 5. 多维审查框架

复杂任务不能只从“能不能跑”一个角度审。

推荐按维度审查：

| 维度 | 关注点 |
|---|---|
| DESIGN | 视觉、交互、信息层级、状态完整性 |
| ROBUSTNESS | 异常、重试、超时、幂等、空态 |
| PERFORMANCE | 渲染、缓存、批量任务、轮询频率、资源消耗 |
| CONVENTION | 目录、命名、架构、项目约定 |
| QUALITY | 可读性、测试、抽象边界、重复代码 |
| SECURITY | 权限、密钥、数据泄露、危险操作 |

不是每次都要展开所有维度。

但高风险任务至少要显式说明：

- 哪些维度已审
- 哪些维度不适用
- 哪些维度待补

## 6. 审查必须能叫停流程

如果审查没有叫停权，它就只是装饰。

以下情况必须停止交付：

- 真相源不明确
- 架构不变量被破坏
- 测试或验证证据不存在
- 关键异常路径未覆盖
- 涉及数据破坏、权限扩大、自动发布但无人工卡点
- AI 明确无法证明结论来源
- 发现“已完成”与实际文件状态不一致

停止不是失败。

停止是避免把不确定性伪装成完成。

## 7. Blocking、Important、Minor

审查意见要分级。

| 等级 | 含义 | 处理方式 |
|---|---|---|
| Blocking | 不修不能进入下一阶段 | 必须修 |
| Important | 不一定阻断，但会明显增加风险 | 优先修或登记债务 |
| Minor | 命名、格式、轻微一致性问题 | 可顺手修或后续处理 |

不要把所有意见都写成 Blocking。

也不要把真正阻断的问题降级成建议。

### 7.1 防止 review 无穷债

review 的目标不是“挑到没话说”。

review 的目标是对固定靶子做可执行裁决。

因此：

- blocking / important 才阻断流程。
- minor 不应驱动反复 loop。
- 可接受的缺口要显式 `accept` 或登记债务。
- 不做的建议要显式 `wontfix`，并说明理由。
- 风格偏好不能伪装成 correctness 问题。

否则 change-review 会变成无底洞，loop 永远无法收敛。

accept / wontfix 的状态语义与裁决权（必须由人裁决，不能由执行 agent 自我豁免）见 §10.2。

### 7.2 严格不是无限加码

严格审查不是把所有问题都卡住。

严格意味着：

- 来源清楚
- 标准清楚
- 证据清楚
- 修法清楚
- verdict 清楚

如果一个问题没有证据、没有影响、没有具体修法，它不应该被写成阻断项。

## 8. 审查证据

审查结论必须尽量有证据。

常见证据包括：

- 文件路径与行号
- 命令输出
- 测试结果
- 截图
- 真机日志
- 网络请求记录
- API 返回样例
- 设计稿节点
- 旧版本行为对比
- 代码引用链

例子：

```text
问题：
实现声称使用生产接口字段 status，但接口样例中字段名是 approval_status。

证据：
- openspec 示例响应字段为 approval_status。
- 当前代码在状态判断处读取 status。
- 测试 fixture 没有覆盖 approval_status，因此没有暴露错误。
```

## 9. 典型纠偏场景

### 9.1 合同变更走错真相源

反模式：

AI 根据旧接口文档修改字段映射，但生产接口已经变化。

纠偏：

- 先确认真相源优先级。
- 生产观测事实高于旧文档。
- 文档作为参考，不作为最终裁决。
- 修改后反向更新 spec 或登记文档债。

抽象规则：

> 接口类改动必须验证当前事实，不能只引用历史文档。

### 9.2 异步失败被静默吞掉

反模式：

任务队列里某一步失败，但 catch 后只打印日志，整体仍显示成功。

纠偏：

- 明确失败是否阻断。
- 阻断失败必须向上抛出或写入状态。
- 自动流程必须有失败通知和复位动作。

抽象规则：

> 无人值守流程里，静默失败等于伪完成。

### 9.3 KDoc 或注释编造引用

反模式：

代码注释写“参考某某规范”，但规范中没有对应条款。

纠偏：

- 删除无法证明的引用。
- 若确实是经验规则，标注为项目规则。
- 重要规则沉淀到 spec 或 AGENTS，而不是藏在注释里。

抽象规则：

> 注释可以解释意图，但不能伪造来源。

### 9.4 子 agent 空跑

反模式：

子 agent 输出“已完成审查”，但没有读取目标文件，也没有证据。

纠偏：

- 主 agent 必须抽查子 agent 的输入和证据。
- 审查报告必须包含文件引用或明确不适用理由。
- 对关键任务，主 agent 保留最终裁决权。

抽象规则：

> 子 agent 可以并行，但责任不能外包。

## 10. 审查记录如何落地

审查记录不是为了留痕而留痕。

它的价值是：

- 让决策可追溯
- 让风险可复盘
- 让经验可回流
- 让相同错误不反复发生

推荐结构：

```markdown
# Review: 任务名称

## Verdict
APPROVE / APPROVE WITH CHANGES / REQUEST CHANGES / RETHINK

## Scope
- 本次审查范围

## Findings
- Blocking
- Important
- Minor

## Evidence
- 文件、命令、截图、日志

## Required Changes
- 必须修复项

## Follow-up Debt
- 可延期但必须登记的问题
```

### 10.1 Pending 与 History

审查记录建议分为两类：

| 类型 | 作用 |
|---|---|
| pending | 当前未归档的审查结果，驱动下一步修复 |
| history | 已归档的审查摘要，用于模式分析 |

典型结构：

```text
plan/reviews/
  change_review_pending.md
  change_review_history.md
```

`pending` 文件保存当前 verdict、issue ID、状态和 must-fix。

`history` 文件保存历史摘要，用于发现重复问题。

注意：**只有 Change Review 需要落盘**。Plan Review 不落盘——它的 must-fix 直接修进计划文档（见 §4.4），计划文档即唯一载体。

### 10.2 审查 issue 状态

每个审查问题都应该有状态。

推荐状态：

- `OPEN`：仍待处理。
- `ACCEPTED`：接受风险，不再阻断。**需要人（而不是执行 agent 自己）裁决**，附理由。
- `WONTFIX`：明确不修，附理由。同样需要人裁决。

问题修复后归档进 history 时记为 `resolved`；`ACCEPTED` / `WONTFIX` 原样归档，保留当时的判断。

状态比单纯删除问题更可靠。

删除会丢失决策，状态能保留判断。

### 10.3 审查记录必须可驱动流程

一份有效审查记录至少能回答：

- 当前能不能继续？
- 不能继续是因为哪些 issue？
- 每个 issue 谁来修、怎么修？
- 修完后如何证明？
- 哪些问题进入债务，不再卡当前流程？

如果审查记录不能驱动下一步动作，它就不是工程资产，只是评论。

## 11. 审查与经验回流

每次审查都应该问一个问题：

> 这个问题是否值得升级成规则？

如果是一次性问题，记录即可。

如果是重复问题，应进入技术债或规则升级候选。

如果是高频高风险问题，应进入：

- AGENTS.md
- spec 模板
- review checklist
- 自动化测试
- lint / script / CI
- skill 或 workflow

审查的终点不是指出错误。

审查的终点是让系统下次更少犯同类错误。

---

审查防的是错误**进入**；错误已经发生（线上 bug、缺陷单）时，走取证式定位流程——见 [`05_诊断方法论.md`](./05_诊断方法论.md)。
