# 05 技术债与经验回流

## 核心定义

技术债不是“代码写得不完美”。

在 AI 协作工程里，技术债指的是：

> 已知存在、暂时不处理、未来会增加理解成本、修改成本、运行风险或治理风险的问题。

经验回流指的是：

> 把一次任务中的失败、审查发现、绕路经验、稳定做法，沉淀回规则、模板、脚本、测试和知识库。

这两者合在一起，构成工程系统的学习能力。

没有技术债登记，AI 会反复踩同一个坑。

没有经验回流，团队只能靠人脑记忆维持质量。

## 1. 为什么 AI 协作更需要技术债台账

人类工程师会记住很多上下文：

- 哪个模块历史包袱重
- 哪个接口文档过期
- 哪个页面有视觉特例
- 哪个自动化脚本不稳定
- 哪类问题已经讨论过

AI 不天然拥有这些长期记忆。

如果债务不显式登记，下一次 AI 很可能：

- 把历史权宜方案当成标准方案
- 删除看似无用但实际有兼容意义的代码
- 重复提出已经否决的方案
- 在旧文档和新事实冲突时选错来源
- 对已知风险没有任何防御

所以 AI 协作里的技术债不是附属文档，而是执行系统的输入。

## 2. 什么需要登记为技术债

不是所有问题都要登记。

技术债登记适用于这些情况：

- 本次明确发现问题，但暂时不修。
- 问题有复发风险。
- 问题会影响后续任务判断。
- 问题会改变架构、业务、接口、测试或发布风险。
- 问题不能靠简单注释解释清楚。
- 问题需要跨任务、跨人、跨 agent 记忆。

常见类型：

| 类型 | 例子 |
|---|---|
| 架构债 | 模块边界临时打穿、状态流不一致 |
| 业务债 | spec 缺字段、异常规则未明确 |
| 接口债 | 文档与生产字段不一致 |
| 测试债 | 只有 happy path，没有异常路径 |
| 数据债 | mock 数据不能覆盖真实状态矩阵 |
| 视觉债 | 关键页面未完成像素级验收 |
| 自动化债 | cron 无幂等、worker 无复位 |
| 文档债 | README 命令失效、旧决策未标废弃 |
| 规则债 | 反复发生的问题仍停留在口头提醒 |

## 3. 技术债记录字段

一条技术债至少包含：

```markdown
## 债务标题

- 类型：
- 发现时间：
- 发现来源：
- 影响范围：
- 现象：
- 风险：
- 触发条件：
- 暂缓原因：
- 当前 workaround：
- 偿还方案：
- 验收标准：
- 状态：
```

字段说明：

| 字段 | 含义 |
|---|---|
| 类型 | 架构、业务、接口、测试、文档、自动化等 |
| 发现来源 | 任务、审查、线上问题、迁移、复盘 |
| 影响范围 | 模块、页面、流程、仓库、工作流 |
| 现象 | 当前看到的问题 |
| 风险 | 不处理会导致什么 |
| 触发条件 | 什么时候会爆发 |
| 暂缓原因 | 为什么现在不修 |
| workaround | 当前临时处理方式 |
| 偿还方案 | 未来怎么消除 |
| 验收标准 | 如何判断债务已还 |
| 状态 | open / in progress / paid / obsolete |

技术债最重要的是说清楚“为什么暂缓”，否则它会变成模糊待办。

## 4. 技术债不是垃圾桶

技术债台账不能变成所有问题的堆放处。

以下内容不应该登记成技术债：

- 立刻能修的小 typo
- 没有影响范围的主观偏好
- 无法复现的问题猜测
- 没有偿还路径的问题抱怨
- 本次任务范围内必须修的 Blocking 问题

Blocking 问题不能通过登记技术债逃避修复。

技术债适用于“可以延期，但必须被看见”的问题。

## 5. 债务分级

推荐按风险分级。

| 等级 | 含义 | 处理方式 |
|---|---|---|
| P0 | 已造成错误结果、数据风险、发布风险 | 立即处理 |
| P1 | 高概率影响后续任务或核心流程 | 纳入近期计划 |
| P2 | 中等风险，影响维护效率 | 排期偿还 |
| P3 | 低风险，主要是清理和一致性 | 批量整理 |

分级时看三个因素：

- 频率：会不会经常遇到？
- 严重性：一旦发生损失多大？
- 修复成本：现在修和以后修差多少？

高频低成本的问题，不应长期登记，应该尽快自动化。

低频高风险的问题，应该进入审查清单和人工卡点。

## 6. 经验回流的四级路径

经验不要只停留在“我知道了”。

推荐四级回流：

```text
一次性发现
  → checklist
  → 稳定规则
  → 自动化约束
```

### 6.1 一次性发现

适合偶发问题。

处理方式：

- 写入审查记录
- 必要时写入 handoff
- 不急着升级成规则

### 6.2 Checklist

适合开始复发的问题。

例子：

- UI 任务必须检查空态、加载态、错误态。
- 接口任务必须确认字段来自当前真相源。
- cron 任务必须有锁和幂等。

### 6.3 稳定规则

适合高频或高风险问题。

规则应写入：

- AGENTS.md
- spec 模板
- review 模板
- 项目 README
- 知识库

例子：

```text
规则：
涉及生产接口字段变更时，不得只依据历史文档。
必须使用当前接口样例、线上观测或可复现请求作为依据。
```

### 6.4 自动化约束

适合可机器检查的问题。

方式包括：

- 单元测试
- e2e
- lint
- schema 校验
- dead key 检测
- screenshot diff
- lock 文件
- queue 状态机
- CI gate

自动化的价值是让同类错误不用再靠人记住。

## 7. 从审查到债务的闭环

审查发现问题后，不只有“修”或“不修”。

正确闭环是：

```text
发现问题
  → 判断是否 Blocking
  → Blocking：立即修
  → 非 Blocking：判断是否有复发风险
  → 有复发风险：登记技术债
  → 多次复发：升级规则
  → 可机器判断：自动化
```

例子：

```text
发现：
多次出现异步任务失败后主流程仍显示成功。

第一次：
审查记录中指出。

第二次：
加入 change-review checklist。

第三次：
沉淀规则：长任务中阻断失败不得静默吞掉。

之后：
增加 worker 状态机测试和失败通知检查。
```

### 7.1 Review history 是模式分析原料

review history 不是归档废纸。

它是识别 recurring pattern 的数据源。

每次 change-review 后，应该把关键摘要归档到 history：

- 日期
- verdict
- issue category
- severity
- 简短描述
- 是否已解决
- 是否建议升级规则

当当前问题和历史问题相似时，应标记为 recurring。

recurring 的价值不是“提醒你又错了”，而是触发系统升级。

### 7.2 三次复发升级规则

推荐使用“三次复发”作为规则升级阈值。

```text
第一次：review finding
第二次：checklist 项
第三次：convention proposal
之后：AGENTS / skill / script / CI
```

这不是机械数字崇拜。

它的意义是防止两种极端：

- 每个偶发问题都升级成规则，导致规则膨胀。
- 高频问题一直停留在人工提醒，导致重复返工。

三次复发后，至少要问：

- 是否应该写入 AGENTS / CLAUDE？
- 是否应该进入 plan-review 或 change-review checklist？
- 是否可以用脚本、lint、测试、CI 自动检查？
- 是否需要做成 skill 或 workflow？
- 是否需要在知识库中补案例？

### 7.3 Convention Proposal

当一个问题具备稳定复发性，应形成规则升级候选。

一条 convention proposal 至少写清楚：

- 规则来源
- 复发现象
- 风险
- 拟升级规则
- 适用范围
- 不适用范围
- 落地位置
- 可自动化部分

例子：

```text
复发现象：
多次出现测试名声称覆盖调用链，但 test body 只检查初始化。

拟升级规则：
测试名与测试体必须契约一致。名称写 invokes / logs / persists / retries 时，测试体必须真实断言对应行为。

落地位置：
AGENTS + change-review checklist。

可自动化：
部分可通过测试命名扫描提示，最终仍需 review 判断。
```

### 7.4 规则升级的落地层级

经验回流不一定都写进同一个地方。

推荐落地顺序：

| 问题类型 | 优先落地位置 |
|---|---|
| 协作流程 | 全局 AGENTS / skill |
| 项目约定 | 项目 AGENTS / CLAUDE |
| 架构铁律 | specs / architecture docs |
| 业务当前态 | openspec / business spec |
| 审查经验 | review checklist |
| 可机器判断 | script / lint / test / CI |
| 暂缓风险 | technical_debt |
| 教学案例 | 案例库 |

稳定规则应进入离执行最近的位置。

能自动化的，不要长期只靠文档提醒。

## 8. 典型债务场景

### 8.1 异步失败静默

现象：

任务执行链里某一步失败，只打日志不改变最终状态。

风险：

- 自动化流程显示完成
- 后续步骤基于错误状态继续
- 人类误以为任务已处理

偿还方案：

- 定义错误等级
- 阻断错误上抛
- 非阻断错误写入状态
- worker 完成前检查失败列表为空

经验回流：

> 所有无人值守流程必须有失败状态、通知和复位路径。

### 8.2 死掉的 i18n key

现象：

翻译 key 已经不再使用，但仍留在资源文件里。

风险：

- AI 误以为功能仍存在
- 搜索结果干扰判断
- 后续清理成本增加

偿还方案：

- 建 dead key 扫描
- 删除前确认无动态拼接
- 记录本次清理范围

经验回流：

> 资源文件不是业务真相源，必须结合引用链判断。

### 8.3 旧文档误导

现象：

接口文档未更新，但代码或生产行为已经变化。

风险：

- AI 按旧字段实现
- 测试 fixture 也跟着旧字段，无法暴露问题

偿还方案：

- 标注文档状态
- 补 current facts
- 更新 openspec
- 审查模板加入“文档新鲜度”检查

经验回流：

> 文档需要状态：current / stale / deprecated / unknown。

### 8.4 伪引用

现象：

注释、KDoc、review 报告声称参考某规范，但无法定位原文。

风险：

- 规则权威性被伪造
- 后续 AI 继续复制错误来源

偿还方案：

- 删除无法证明的引用
- 可保留经验判断，但标为 local rule
- 高价值规则写入知识库

经验回流：

> 来源不可验证的规则，不得包装成上层规范。

### 8.5 视觉资产缺失

现象：

页面缺图，AI 用占位图或临时图替代。

风险：

- 视觉验收失真
- 设计还原被错误认为完成

偿还方案：

- 先搜索上游项目、设计稿、资源目录
- 找不到再登记资产债
- 明确占位范围和替换条件

经验回流：

> 缺资产不是自由发挥的授权，先找源，再占位。

## 9. 技术债偿还

技术债必须定期清理。

可采用三种方式：

### 9.1 顺手偿还

当任务正好触达相关区域，且风险可控，顺手偿还。

要求：

- 不扩大任务边界过多
- 有验证
- 在交付说明中写清楚

### 9.2 批量偿还

适合同类债务。

例子：

- 清理一批 dead key
- 统一一批接口字段
- 修复一批过期 README 命令
- 为多个 worker 加失败状态检查

要求：

- 先做批任务拆解
- 有统一验收标准
- 分批提交

### 9.3 专项偿还

适合架构债、状态机债、自动化债。

要求：

- 单独计划
- Plan Review
- 明确迁移路径
- 明确回滚方式
- Change Review

## 10. 技术债台账维护原则

技术债台账应该保持可读、可行动。

维护原则：

- 新增债务要有触发条件和偿还路径。
- 已偿还债务要标记 paid，而不是直接删除。
- 已失效债务标记 obsolete。
- 重复债务合并。
- 长期未处理的 P1/P0 必须重新评估。
- 如果债务变成规则，链接到规则位置。
- 如果规则变成自动化，链接到脚本或测试。

技术债台账不是历史垃圾堆。

它应该是 AI 执行前会读取的风险地图。

## 11. 经验如何进入知识库

每次任务结束后，可以问五个问题：

- 这次哪里发生了漂移？
- 哪个判断依赖了隐性经验？
- 哪个问题如果早有模板就能避免？
- 哪个检查可以自动化？
- 哪个规则应该上升到上层定法？

经验进入知识库时，不要只写故事。

推荐结构：

```text
经验来源：
发生了什么：
错误做法：
正确做法：
抽象规则：
适用边界：
可自动化部分：
```

知识库的价值不是记录“当时怎么做”。

它的价值是让未来 AI 和人类在相似情境下更快做对。

## 12. 从经验到 skill

当某类流程反复出现，仅有模板还不够。

它应该被封装成 skill、脚本或 workflow。

适合封装的信号：

- 步骤稳定
- 输入输出稳定
- 需要严格顺序
- 经常被 AI 漏步骤
- 需要生成固定格式产物
- 需要读取固定规则源
- 需要写入固定归档位置

例如：

- plan-review：固定读取计划、挑战设计、输出 verdict。
- change-review：固定读取 diff、关闭 plan-review、跑 lint、写 pending。
- commit-flow：固定归档 review、核对 diff、提交。
- cron-handoff：固定生成续作交接。

模板让人知道怎么做。

skill 让 AI 每次按同一条轨道做。

CI 和脚本让系统在 AI 忘记时仍然拦住。
