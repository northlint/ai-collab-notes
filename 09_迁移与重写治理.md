# 09 迁移与重写治理：fork、绿地与 parity 对账

## 核心定义

迁移和重写是 AI 参与度最高、也最容易失控的一类工程：源材料多而互相矛盾、目标是"和旧的一致"这种移动靶、工作量大到必然要批量和并行。

《下层执行》管的是拆解与编排（Phase 0/1/2、批次、并行边界）；本章管的是迁移特有的**源治理与收口**：拿什么当真相、怎么证明"一致"、什么时候可以宣布旧源退役。

## 1. 两种模式：fork 治理 vs 绿地重写

| 维度 | Fork 治理 | 绿地重写 |
|---|---|---|
| 做法 | 快照分叉旧仓，在其上改造演进 | 新仓从零建，旧实现只作参照 |
| 适合 | 旧代码质量可接受，改造面 < 重写面 | 旧代码是反面教材，架构要换代 |
| 核心风险 | 旧包袱与新规则长期共存、界限模糊 | 参照源被"顺手照抄"，旧坏模式渗入新仓 |

判断标准不是感情（"旧代码好烦"），而是账：**需要保留的逻辑量 × 改造成本 vs 重写成本 + 对账成本**。

### 1.1 Fork 治理的纪律

- **快照分叉，显式声明不跟踪上游**：分叉即独立演进；如果还要同步上游，那不是 fork 是分支，纪律完全不同。
- **登记分叉基线**：从哪个版本、哪个 commit 分叉，写进项目文档——未来所有"这段代码为什么这样"的考古都从这里起步。
- **改造清单显式化**：包名、命名前缀、依赖替换等全局改造，列成清单逐项执行并验证，不散落在各次提交里。
- **历史文档标注状态**：带过来的旧文档逐份标 current / historical——AI 会把历史当现行（真相源误用的高发区）。

### 1.2 绿地重写的纪律

- **搬契约，不搬结构**：旧实现里值得带走的是接口路径、字段语义、状态机、业务规则这些**契约**；目录结构、组件划分、实现套路按新架构重来。照抄结构 = 把旧债搬进新仓。
- **冻结参照源**：对账期间参照实现不再变化（或锁定版本），否则"一致"是移动靶。
- **禁止顺手复制**：从参照源"参考"和"粘贴"是两件事；lint / review 里显式设"不抄旧代码"的检查项。

## 2. 多源分维度授权

迁移现场常见四五个源同时存在：旧生产实现、接口文档、设计稿 / 原型、产品逻辑说明、新架构蓝图。真相源治理（见《上层定法》§3）在迁移里的具体化是**分维度授权**——每个维度恰好一个权威：

| 维度 | 权威源 | 理由 |
|---|---|---|
| 接口路径 / 方法 / 信封 / 接口存废 | **实跑通的生产实现** | 它每天在真实流量下工作 |
| 字段类型 / 可空性 | **接口文档**（据后端代码生成的） | 生产前端可能容错掩盖了真实类型 |
| 视觉 | 设计 token / 原型 | 生产实现的视觉可能已经过时 |
| 行为 / 状态机 / 校验 | 产品逻辑说明（PRD）；被它简化的深逻辑回查生产实现 | 文档管意图，实现管细节 |
| 结构 / 技术 | 新架构蓝图 | 它沉默或编不出 → **问人，不擅自定** |

冲突不问"哪个源更可信"，先问"冲突在哪个维度"，按表分流；表答不了的，标记阻塞回问，不脑补。

## 3. Parity 对账：先审计，再修复

"迁移完了"最危险的版本是"我觉得都搬过去了"。可靠的做法是把"一致性"变成一份**可检查的清单**，分两步走：

### 3.1 审计：产出差异清单

- 逐页 / 逐模块对照参照源，把差异逐条登记：缺失的逻辑、行为不一致、字段处理不同、有意简化。
- 审计阶段**只记录不修复**——边审边修会让审计永远做不完，且覆盖面无法证明。
- 每条差异标注：来源证据（参照源的位置）、严重级别、处理方式（本批修 / 分批修 / 登记债务 / 有意不做）。

### 3.2 修复：按清单分批销账

- 差异清单就是批任务清单，按《下层执行》的拆批纪律执行。
- 每批修复后在清单上销账；清单归零（或余项全部显式转为技术债 / 有意不做）= parity 达成。
- "有意不做"必须写理由——它是决策，不是遗漏。

## 4. 收口：外部源退役

迁移完成的标志不是"新的能跑"，而是**外部源退役**：

- [ ] 参照源、旧文档在新仓文档里的引用逐处摘除或标注 historical。
- [ ] 为迁移期设置的"必读文档集"瘦身——对账用的源裁决规则、参照指引可以退场（见《上下文治理》必读矩阵）。
- [ ] 迁移期专用的 lint / 检查项评估去留。
- [ ] 从此新仓是唯一真相源，旧源不再被任何流程读取。

不收口的迁移会留下永久的双源状态：AI 一半时间读新仓、一半时间读旧参照，真相源治理前功尽弃。

## 5. 无人值守迁移的额外前提

迁移是无人值守的高适配场景（重复、边界清晰、有参照），但必须满足《长任务、Loop、Cron 与无人值守》§6 的准入条件，特别是：

- 参照源已冻结（否则追移动靶）；
- 有可自动判断的 oracle（参照对照、状态矩阵、schema 校验、screenshot diff——反面案例见《案例库》#10）；
- 差异清单已存在——无人值守销账可以，无人值守**审计**不行，审计的覆盖面判断需要人复核。

## 6. 迁移治理检查清单

- [ ] 模式已选（fork / 绿地），判断依据是成本账不是情绪？
- [ ] fork：分叉基线已登记、不跟踪上游已声明、旧文档已标状态？
- [ ] 绿地：参照源已冻结、"搬契约不搬结构"已成检查项？
- [ ] 多源分维度授权表已建立，冲突分流规则明确？
- [ ] parity 走"先审计出清单、再分批销账"，不是边看边修？
- [ ] 差异清单每条有证据、级别、处理方式？
- [ ] 收口计划存在：外部源退役 + 必读集瘦身？
