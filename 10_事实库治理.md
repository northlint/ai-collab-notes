# 10 事实库治理：系统当前态的事实沉淀与分诊入口

## 核心定义

治理线的其他章节管"法"与"流程"：架构不变量说系统**应该**怎么搭，业务 Spec 说用户**应该**能做什么，计划说这次要改什么。还有一类容易被漏掉的知识资产：**系统实现事实上是什么样**——哪个业务名词对应哪个字段，哪个页面对应哪个组件，哪类症状的根因在哪一层。

事实库（kb）承载的就是这类知识：

> 沉淀"系统本来是什么样"的事实型知识，让 AI 和人拿到一个 bug 或一条需求时，一两次 grep 就定位到代码，而不是每次全量扫描。

与三个邻居的分工必须先划清（另见 [`01_上层定法.md`](./01_上层定法.md) §2 计划与 Spec 的区别）：

| 资产 | 回答的问题 | 性质 |
|-|-|-|
| 计划 / openspec | 这次要改什么 | 变更流 |
| 业务 Spec | 用户应该能做什么、系统应如何表现 | 应然当前态 |
| 工作规范 / skill | 代码该怎么写、流程怎么跑 | 操作规范 |
| **事实库** | 实现事实上长什么样、症状怎么走到代码 | **实然当前态** |

事实库只放事实：计划与规范不进来，未验证的猜测也不进来——猜测进计划流讨论，证实后才落库。

## 1. 为什么值得单独治理

诊断方法论（见 [`05_诊断方法论.md`](./05_诊断方法论.md)）的维度②"定位锚点"，是每个 bug 都要付一遍的成本。没有事实库时：

- AI 每次从零 grep，读完整个目录找一个按钮；
- 同类 bug 每次重新定位，同一条路径被反复购买；
- 新会话、新人、新 agent 重复摸索同一套机制；
- 更糟的是把"没找到"当"不存在"，或把资源残留当成功能存在。

事实库做的事，是把这类**重复侦察**沉淀成资产：诊断是查一次，事实库是把查的路径固化下来。它和上下文治理（见 [`02_上下文治理.md`](./02_上下文治理.md)）是同一笔账的两面——02 管"规则如何低成本到达 AI"，本章管"事实如何低成本到达 AI"。

## 2. 归属规则：知识跟着它描述的对象走

- **单一仓库的事实，放进该仓自己的 `kb/` 目录**：随代码提交、clone 即得、天然与代码同版本——"改代码顺手更新"只有同仓时才是零摩擦的。
- 只有"放进任何单一仓库都不对"的知识才外置：
  - **跨分支**：一个仓库多条分支承载多套系统，事实按分支划分，放任一分支都会误导其他分支；
  - **跨仓共享**：多个仓库共用的产品线术语、跨仓协议。
- 判断标准一句话：

> 这条知识写进任何单个仓库都会"放错地方"，才允许外置。

- 新项目从统一模板复制骨架；分类文件名跨项目**不改名**（见 §4），换项目零学习成本。

## 3. 入口设计：README 是分诊路由器，不是文件清单

事实库最常见的失败不是内容错，而是入口错：README 写成文件清单（"本目录包含以下文档……"），等价于要求读者全读——知识库反而变成每次任务的固定阅读税（案例库 #25）。

入口的正确形态是**分诊**。

### 3.1 分诊第一刀

拿到 bug 先问一句：**换个页面 / 模块会不会也这样？**

- **会 → 横切类**：根因在某层公共机制，一处修复通吃多页，先找页面反而绕远 → 走轴一。
- **只有这一处 → 单点类**：具体功能行为不符预期，要先定位到组件 → 走轴二。

### 3.2 轴一：症状 → 机制表

| 列 | 作用 |
|-|-|
| 症状 | 用用户语言描述（导出乱码 / 频繁被踢 / 文案不对） |
| 根因层 | 请求层 / 鉴权 / i18n / 权限……直接到层 |
| 具体位置 | file:line，**必须 grep 核实过** |
| 范围 | **全局一处**（直接去改）/ **每页模式**（还需轴二定页） |

"范围"列是这张表的治理关键：把只在一处成立的口径写成全局机制，修复就会走错层、波及无辜（案例库 #26）。

### 3.3 轴二：按"抓手"由省到贵排序

按 bug 描述里给的线索排定位手段，越靠前越便宜：

1. **界面原文**（按钮字 / 提示语）→ 两跳 grep：语言包拿 key → key 定位组件；无 i18n 的直接原文一跳。
2. **一个操作 / 一段流程 / 跨页联动** → 用户旅程（user-journeys）。
3. **只有业务名词** → 术语表（domain-glossary）。
4. **已知路由 / 页面路径** → 代码地图（code-map）。
5. 都不中 → 全仓 grep 兜底，或顺 journey 依赖链找。

两条硬纪律：

- 入口里写的每条 grep 路径，必须**用一个真实案例跑通过**再写进去，附示例——没跑通的定位路径是伪事实。
- 入口显式写上反指令："**不要把文档全读一遍再动手**"——那正是事实库要消灭的全量扫描。

这套入口本质上是必读矩阵思想（见 [`02_上下文治理.md`](./02_上下文治理.md) §4）在事实层的应用：按任务特征路由，而不是全量加载。

## 4. 文档结构：固定四件套 + 固定文件名

| 文件 | 管什么 | 防什么 |
|-|-|-|
| architecture | 技术栈 / 模块划分 / 关键机制细节（轴一症状表在此展开） | 机制散落各处 |
| code-map | 路由 ↔ 组件 ↔ API 总索引 + 命名陷阱 | "从名词猜不到位置" |
| domain-glossary | 业务术语 → 字段 / 枚举 / 模块 | 名词与代码脱节 |
| user-journeys | 命名业务闭环 + 每步操作 → 方法名 | 动作定位与跨页依赖无处可查 |

- **文件名跨项目统一不改名**：治理价值在心智一致——任何项目的事实库都长一个样，人和 AI 换项目不用重新学结构。
- user-journeys 单独成文件的理由：**动作定位**（描述一个操作 → 方法名）和**跨页数据依赖**（A 配了 B 没生效）是其他索引结构性覆盖不到的两类线索。
- user-journeys 必须防膨胀，四条边界：
  1. 只收"命名的业务闭环"（测试：这个流程 PM 会给它起名吗？）；
  2. 条数封顶（约 5–15 条）；
  3. 只写步骤顺序 + 关键分支，函数级细节归 code-map / domain-glossary；
  4. 排除异常 / 边缘 / AB 分支，只留 happy path + 至多 1–2 个关键决策分支。

## 5. 物证纪律与可信度

事实库的可信度靠三层机制维持：

- **写入门槛**：只写已验证事实。file:line 须 grep 核实；方法名逐一核实真实存在；只做了同构推断、没逐一展开的，如实标注，不假装全验证。
- **不确定性显式化**：无法核实的（如路由由后端动态下发）标"待确认"，并给出可信的替代锚点（组件文件路径可信，就以它为准）——别让一个未确认字段拖累整块索引的可用性。
- **保鲜机制**：每篇头部标注 `适用范围 + 最后校核日期`；引用时按校核日期判断可信度，过期先核对再采信。**改代码时发现事实库过时或缺失 → 顺手更新；改了代码没更事实库 = 没改完**（与文档同步是同一条纪律，见 [`07_工具链治理.md`](./07_工具链治理.md) §1.1）。

## 6. 负知识：确认不存在也是资产

事实库应设一段"**已排查确认不存在的功能**"：

- 记录结论 + **排查依据**（搜过哪些关键词、看过哪些目录）；
- 价值一：防重复寻找——不落盘的排查会被下个会话原价重买（案例库 #27）；
- 价值二：防误判存在——资源残留（死 i18n key、旧配置）会让 AI 以为功能存在，负知识是解毒剂（另见案例库 #11 mock、《技术债与经验回流》§8.2 死 key）；
- 负知识同样要校核日期："不存在"也会过期（后来加上了）。

## 7. 填充与核查：省钱在检索，保质在核查

建库 / 补库是典型批任务，套用下层执行的模型经济学（见 [`03_下层执行.md`](./03_下层执行.md) §5.4）：

- **检索广撒网**：多个降档模型的检索型子 agent 并行探路（扫路由、扫机制、扫术语、扫 journey），产出候选事实；
- **核查者不降级**：核查者的职责是**证伪**——逐条 grep 核实 file:line、真实跑通入口示例、核对范围列（全局一处还是每页模式），未核实的不定稿；
- 多目标（多仓 / 多分支 / 多品牌）时按目标拆核查者并行，检索结果不跨目标混用。

定稿的每条事实都应答得出"哪次工具动作支撑它"——和诊断的证据回指（见 [`05_诊断方法论.md`](./05_诊断方法论.md) §1.3）同一条纪律。

## 8. 与诊断的闭环

事实库和诊断方法论互为供需：

- **诊断消费事实库**：六维侦察的维度①②先走分诊入口，命中则一两次 grep 得锚点；
- **诊断反哺事实库**：每次诊断收尾问一句——这次的定位路径事实库覆盖了吗？没覆盖的，把症状行 / journey / 负知识补进去。这是经验回流（见 [`06_技术债与经验回流.md`](./06_技术债与经验回流.md)）在"定位类事实"上的具体通道。

## 9. 事实库治理检查清单

- [ ] 只有已验证事实？计划、规范、猜测各回各家？

- [ ] 每篇头部有适用范围 + 最后校核日期？

- [ ] 归属正确：单仓知识在仓内 `kb/`，外置的确属跨分支 / 跨仓？

- [ ] 入口是分诊路由器而非文件清单？有"别全读"的显式反指令？

- [ ] 症状表有"范围"列，区分全局一处 / 每页模式？

- [ ] 写进入口的定位路径都用真实案例跑通过？

- [ ] 待确认项有标注 + 替代锚点？

- [ ] 有负知识段（结论 + 排查依据）？

- [ ] 文件名与模板一致，没有自创命名？

- [ ] "改代码顺手更新事实库"有执法落点（review checklist）？
