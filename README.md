# AI 协作经验

> 主题：为 AI 协作建立一套"上层定法、下层执行、审查纠偏、经验回流"的工程治理系统，并补齐上下文治理、诊断、工具链治理，以及长任务、loop、cron、批任务、无人值守等执行编排方法。

## 这套知识库是什么

这不是一组"AI 写代码技巧"，也不是某个项目的专属手册。

它沉淀的是一套通用工程方法：当 AI 参与长期软件工程时，如何把项目从"代码仓库"升级成一个 AI 可以正确操作、可审查、可回滚、可持续演进的治理系统。

核心判断：

- AI 的问题不只是"会不会写代码"，而是会越界、漂移、脑补、自证完成、跨会话遗忘。
- 解决这些问题不能靠 AI 自觉，而要靠上层约束、流程轨道、对抗审查、机械验证和经验回流。
- 长任务、无人值守、批量迁移、自动修 bug 等场景，还需要专门的执行编排系统：loop、cron、watch、worker、队列、人工卡点、oracle、自验证、handoff。

## 定位声明

- 本知识库是**方法论源**：讲原理、设计理由和适用边界（why），自包含、可迁移，不依赖任何私有工具。
- 各机制在你的环境里落地成 skill / 命令 / 脚本后，**运行时以落地实现为准**；本知识库与落地实现保持**机制级一致**（verdict 枚举、状态语义、落盘契约、闭环走向），措辞与清单条目级差异不算脱节。
- 落地实现的机制变更后，对应章节欠一次同步——同步纪律本身应配机械触发物（如哈希基线比对），见《07 工具链治理》。
- **主入口是飞书知识空间「AI协作经验」**，可以直接在飞书里浏览和编辑；本仓库的 markdown 是编辑时的工作副本 + 本地恢复兜底，不假设本地文件永远反映最新状态——改动前建议先跟飞书那边核对。跨章节链接在本地是相对路径（`./0X_xxx.md`），发布到飞书时会替换成飞书文档的绝对链接，本地文件本身不受影响。

## 总框架

整套体系分为两条主线。

### 治理线

治理线回答：AI 应该在什么规则下工作、规则如何高效到达 AI、谁说了算、做错了怎么发现、错误怎么定位、经验怎么沉淀、约束由谁执法。

```text
上层定法
  → 上下文治理（规则如何到达 AI）
  → 下层执行
  → 审查纠偏（防错误进入）
  → 诊断方法论（查已发生的错）
  → 技术债与经验回流
  → 工具链治理（规则升级为机器执法）
  → 回到上层定法
```

### 执行线

执行线回答：AI 如何完成长任务、批任务、自动续作、无人值守作业，以及迁移/重写这类高危批量工程如何治理。

```text
任务进入
  → 队列 / 触发器
  → 计划 / 分批 / 分相
  → 执行器
  → 自验证（边做边验）
  → 检查点 / 人工卡点
  → 交付 / 归档 / 续作
```

## 核心概念索引

| 概念 | 说明 |
|---|---|
| 架构不变量 | 项目的"宪法"，定义系统怎么搭、哪些红线不可破 |
| 业务 Spec | 用户能力的当前态，区别于一次性的计划文档 |
| 真相源治理 | 规定业务、视觉、契约、架构、运行事实分别由谁管 |
| specs / openspec | specs 管工程宪法，openspec 管业务变更流 |
| 上下文经济 | 注意力是稀缺资源；常驻规则的门槛是"每次都要 + 别处没有" |
| 薄宪法 | 常驻规则文件只留要义 + 指针，全文与台账沉专用文档 |
| 必读矩阵 | 必读文档按任务类型路由，不全量 ⭐ |
| Workflow | 把 AI 操作放进轨道，减少自由裁量 |
| Plan Review | 执行前审查计划，防止方向错、源错、范围错 |
| Change Review | 改动后审查实现，防止假完成、漏验证、破坏约束 |
| 检查点协议 | 产出后必停、verdict 驱动流转；无人值守时唯一自动分支 = APPROVE |
| 诊断六维 | 现象确认 / 锚点 / 根因上溯 / 爆炸半径 / 横向兄弟 / 历史意图 |
| Review History | 审查历史沉淀，用于识别重复模式并升级规则 |
| 技术债 | 已知未完成项的显性账本，允许暂缓，不允许遗忘 |
| 治理债 | 治理机制自身的死代码；防的问题能在更源头零成本消掉的机制该废弃 |
| 经验回流 | 从单次问题到 recurring pattern，再升级为规则或门禁 |
| 执法层级 | 模型自律 < 文档 < review < 机器闸门 < harness 硬拦；升级规则 = 升层级 |
| verdict 物证化 | 自动推进只认落盘物证，口头声称无效 |
| 模型分层 | 省钱在检索，保质在核查；输出直接决定 blocking verdict 的不降级 |
| Loop Workflow | 以 oracle、门禁、自检、审查和棘轮构成的可收敛闭环 |
| Cron | 定时触发机制，适合兜底、续作、巡检，不适合所有实时任务 |
| Watch | 轻量看守进程，发现事件后退出并唤醒主会话 |
| Worker | 可串行处理完整任务链路的执行器 |
| Oracle | 无人值守任务中的对错神谕，如活实现、mock 矩阵、截图对比 |
| 假绿 | 门禁或自检看似通过，但实际没有验证到目标；闸门须自证执行 |
| 传感器校准 | 确认截图、设备、缓存、测试、API 等验证工具看到的是当前事实 |
| Parity 对账 | 迁移一致性 = 先审计出差异清单，再分批销账，最后外部源退役 |
| Park-don't-guess | 不确定时停下入队列，不靠猜补 |

## 使用方式

### 作为方法论阅读

按章节顺序阅读，建立完整心智模型：

```text
00 → 01 → 02 → 03 → 04 → 05 → 06 → 07（治理线一圈）
→ 08 → 09（执行编排与专题）
A / B 按需查阅（附录）
```

### 按问题类型进入

| 你在做什么 | 读 |
|---|---|
| 设计项目治理 | 01 上层定法 |
| 控制规则/文档的加载成本 | 02 上下文治理 |
| 安排一次复杂任务 | 03 下层执行 |
| 做质量控制 | 04 审查纠偏 |
| 排查一个 bug | 05 诊断方法论 |
| 沉淀经验 / 清理机制 | 06 技术债与经验回流 |
| 建设 skill / 闸门 / 钩子 | 07 工具链治理 |
| 设计自动化或长任务 | 08 长任务与无人值守 |
| 做迁移 / 重写 | 09 迁移与重写治理 |
| 给别人解释 | A 案例库 |
| 直接执行 | B 模板库 |

### 作为项目治理初始化清单

新项目或现有项目接入 AI 协作时，优先检查：

- 是否有架构不变量
- 是否有业务 Spec 或当前态能力文档
- 是否定义了真相源和冲突裁决
- 是否有计划、审查、技术债、经验回流流程
- 常驻规则是否满足"每次都要 + 别处没有"（薄宪法）
- 是否有长任务和无人值守边界

## 维护原则

- 本知识库沉淀通用方法，不写死某个项目的路径、命令、技术栈。
- 具体项目可以引用这里的方法，再在自己的项目文档里落具体实现。
- 案例可以来自真实项目，但要抽象成通用规则。
- 重复出现的问题应从案例进入规则候选，再进入模板或自动化门禁。
- 这套知识库本身也应接受经验回流：如果某类问题反复出现，更新对应章节。
- **同步纪律**：工作流规则 / 落地 skill 的机制变更时，同步修订对应章节；冲突以落地实现为准，视为本库欠一次同步（机制级一致即可，见「定位声明」）。
- 章节顺序的权威是本 README 索引。**双命名空间**：主线章节（`00`–`09`）数字前缀镜像 README 顺序，插入新主线章节时对受影响区间做一次机械重编号；案例库 / 模板库是纯附录，用字母前缀（`A_`/`B_`）**独立命名空间**，不随主线插章变化，永远不需要重编号。
