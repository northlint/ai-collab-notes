# AI 协作工程治理知识库

> 主题：为 AI 协作建立一套“上层定法、下层执行、审查纠偏、经验回流”的工程治理系统，并补齐长任务、loop、cron、批任务、无人值守等执行编排方法。

## 这套知识库是什么

这不是一组“AI 写代码技巧”，也不是某个项目的专属手册。

它沉淀的是一套通用工程方法：当 AI 参与长期软件工程时，如何把项目从“代码仓库”升级成一个 AI 可以正确操作、可审查、可回滚、可持续演进的治理系统。

核心判断：

- AI 的问题不只是“会不会写代码”，而是会越界、漂移、脑补、自证完成、跨会话遗忘。
- 解决这些问题不能靠 AI 自觉，而要靠上层约束、流程轨道、对抗审查、机械验证和经验回流。
- 长任务、无人值守、批量迁移、自动修 bug 等场景，还需要专门的执行编排系统：loop、cron、watch、worker、队列、人工卡点、oracle、自验证、handoff。

## 总框架

整套体系分为两条主线。

### 治理线

治理线回答：AI 应该在什么规则下工作，谁说了算，做错了怎么发现，经验怎么沉淀。

```text
上层定法
  → 下层执行
  → 审查纠偏
  → 技术债与经验回流
  → 规则升级
  → 回到上层定法
```

### 执行线

执行线回答：AI 如何完成长任务、批任务、自动续作、无人值守作业，以及如何在高自治和人类卡点之间取得平衡。

```text
任务进入
  → 队列 / 触发器
  → 计划 / 分批 / 分相
  → 执行器
  → 自验证
  → 人工卡点
  → 交付 / 归档 / 续作
```

## 核心概念索引

| 概念 | 说明 |
|---|---|
| 架构不变量 | 项目的“宪法”，定义系统怎么搭、哪些红线不可破 |
| 业务 Spec | 用户能力的当前态，区别于一次性的计划文档 |
| 真相源治理 | 规定业务、视觉、契约、架构、运行事实分别由谁管 |
| specs / openspec | specs 管工程宪法，openspec 管业务变更流 |
| Workflow | 把 AI 操作放进轨道，减少自由裁量 |
| Plan Review | 执行前审查计划，防止方向错、源错、范围错 |
| Change Review | 改动后审查实现，防止假完成、漏验证、破坏约束 |
| Review History | 审查历史沉淀，用于识别重复模式并升级规则 |
| 技术债 | 已知未完成项的显性账本，允许暂缓，不允许遗忘 |
| 经验回流 | 从单次问题到 recurring pattern，再升级为规则或门禁 |
| Loop Workflow | 以 oracle、门禁、自检、审查和棘轮构成的可收敛闭环 |
| Cron | 定时触发机制，适合兜底、续作、巡检，不适合所有实时任务 |
| Watch | 轻量看守进程，发现事件后退出并唤醒主会话 |
| Worker | 可串行处理完整任务链路的执行器 |
| Oracle | 无人值守任务中的对错神谕，如活实现、mock 矩阵、截图对比 |
| 假绿 | 门禁或自检看似通过，但实际没有验证到目标 |
| 传感器校准 | 确认截图、设备、缓存、测试、API 等验证工具看到的是当前事实 |
| Park-don't-guess | 不确定时停下入队列，不靠猜补 |

## 使用方式

建议按三种方式使用。

### 作为方法论阅读

按章节顺序阅读：

```text
00 → 01 → 02 → 03 → 04 → 05 → 06 → 07
```

适合建立完整心智模型。

### 作为项目治理初始化清单

新项目或现有项目要接入 AI 协作时，优先检查：

- 是否有架构不变量
- 是否有业务 Spec 或当前态能力文档
- 是否定义了真相源和冲突裁决
- 是否有计划、审查、技术债、经验回流流程
- 是否有长任务和无人值守边界

### 作为执行模板库

实际做事时直接使用 `模板库.md`：

- 做计划时复制计划模板
- 做审查时复制 review 模板
- 暂缓事项复制技术债模板
- 要做自动续作或无人值守时复制 loop / cron / worker 模板

## 维护原则

- 本知识库沉淀通用方法，不写死某个项目的路径、命令、技术栈。
- 具体项目可以引用这里的方法，再在自己的项目文档里落具体实现。
- 案例可以来自真实项目，但要抽象成通用规则。
- 重复出现的问题应从案例进入规则候选，再进入模板或自动化门禁。
- 这套知识库本身也应接受经验回流：如果某类问题反复出现，更新对应章节。
