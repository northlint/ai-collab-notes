# 飞书知识库迁移计划

> 状态：**待 review**（讨论已收敛，尚未执行）

## 1. 背景与目标

现状：这份 KB 纯本地 markdown + git，本地为唯一副本，同事无法方便浏览，也没有正式的分享/协作入口。

目标：迁移到飞书知识库（wiki），作为**主要的阅读和编辑入口**——不是导出快照，是可以随时直接点开编辑的活文档；本地文件退化为 AI 编辑时的工作副本 + 出问题时的静默恢复兜底。

## 2. 已锁定的架构决策（讨论过程见对话记录）

| 决策点 | 结论 | 理由 |
|---|---|---|
| 文档载体 | 飞书原生 docx（`lark-doc`），不是 Drive 里的 markdown 文件类型 | 要的是"真正的知识库文档、能直接点开编辑"的体验，不是技术化的文件预览 |
| 编辑权限 | 仅用户本人 | 无并发冲突场景；"写前 fetch"降级为轻量习惯而非硬闸门 |
| 本地 git 角色 | 保留，降级为**静默恢复兜底** | 不再对这份 KB 的改动走 `/commit` 检查点仪式；git commit 动作本身可以继续做，但不再需要暂停问用户 |
| 目标知识空间 | 新建一个专属知识空间 | 和用户其他私人文档边界清晰，以后分享给同事只需开这一个空间的权限 |
| 节点树结构 | 全部平铺一层，用 `00`-`09`/`A`/`B` 标题前缀排序 | 和本地文件结构完全对应，最简单，以后加章节不用调整层级 |
| 链接重写精度 | 第一版只做**文档级**跳转（章节间链接指向对方文档首页） | 锚点级需要额外一轮 `fetch --detail with-ids` 找每个引用锚点的 `block_id`，且 `overwrite` 覆盖后 block_id 可能变化、需要重新验证，工作量和维护成本明显更高，先不做 |
| 迁移范围 | 直接批量迁移全部 13 个文档（不做 1-2 章试点） | 机制已通过 lark-cli 文档核实（非猜测），用户接受"如有意外全部 13 个文档返工"的风险 |

## 3. 技术机制（已用 lark-cli skill 文档核实，非猜测）

- **建**：`lark-cli docs +create --api-version v2 --doc-format markdown --parent-token <知识空间/父节点 token> --content <本章 markdown 全文>`
  一步直达目标知识空间位置，不需要先 `wiki +node-create` 再搬运。
- **建空间**：`lark-cli wiki +space-create`（仅 user 身份）新建专属知识空间，拿到 `space_id` 供后续 `docs +create --parent-token` 使用。
- **读**：`lark-cli docs +fetch --api-version v2 --doc <token>` —— 执行时需确认输出格式参数（默认可能是 XML，需要看是否有 markdown 输出选项，如果没有则退化为读 XML 再人工/工具转换比对，这是执行阶段要落实的一个细节，不影响整体方案）。
- **写**：`lark-cli docs +update --api-version v2 --doc <token> --command overwrite --content <新 markdown 全文>` —— **原地覆盖**，`document_id`/`url` 不变。
- **关键事实**：`+create` 建一次后 token 永久稳定；后续只用 `+update overwrite`，不会像 `drive +import` 那样每次生成新文档、造成链接失效。

## 4. 迁移步骤

1. `lark-cli wiki +space-create` 新建专属知识空间，记录 `space_id`。
2. 按 `00_总纲.md` → `09_迁移与重写治理.md` → `A_案例库.md` → `B_模板库.md` → `README.md` 的顺序，逐个 `docs +create --parent-token <space_id> --doc-format markdown --content @<本地文件>`，记录每个返回的 `document_id`/`url`。
3. 全部建完后，做一次**文档级链接重写**：本地 markdown 里的 `[文本](./02_xxx.md#锚点)` 形式的相对链接，替换成对应飞书文档的 `url`（锚点部分丢弃，因为本版只做文档级）。重写后的内容通过 `docs +update --command overwrite` 写回每个文档一次。
4. 更新本地 `README.md`：加一节说明"本 KB 的主入口是飞书知识库 <空间链接>，本地文件是 AI 编辑时的工作副本"，避免以后有人（包括我自己）把本地当权威源。
5. 更新 `plan/README.md` 的"分发声明（D4）"：原声明是"对外分发物 = 本地正文 markdown"，现在要改成"主分发/协作入口 = 飞书知识空间；本地 markdown 是编辑时的工作副本与恢复兜底"。
6. 新建一个本地专属映射表 `plan/feishu_node_map.md`（记录 13 个文件 ↔ `document_id`/`url` 的对应关系），供以后 AI 编辑时查表，不作为对外分发物。
7. 抽查验证：至少 3 个文档打开确认渲染正常（标题层级、代码块、表格、列表），至少 3 处跨文档链接确认可点击跳转。

## 5. 迁移完成后的日常工作流（替代原来的"改本地文件→`/commit`"）

1. 开始编辑某一章前，`docs +fetch` 一次刷新本地工作副本（轻量习惯，防止漏掉自己上次在飞书 App 里随手改的内容）。
2. 本地正常用 Read/Edit/Grep 编辑、跑既有的检查（专名泄露 grep、内部链接有效性等）。
3. 改完后 `docs +update --command overwrite` 写回飞书，本地 git 照常 commit 一次（不再暂停问 `/commit` 三选一，直接推进）。
4. 不再维护 `check_sync.sh` 那套"比对本地 skill 文件 md5"的机制对这份 KB 本身的适用性不变——那个脚本管的是"本 KB 与另外 4 个 live skill 文件"的同步，和这次的"本地↔飞书"同步是两件事，互不影响，无需改动。

## 6. 验收标准

- **V1**：13 个文档全部创建成功，`plan/feishu_node_map.md` 记录完整的 token/url 映射。
- **V2**：抽查渲染无误——标题层级、代码块、表格、列表符号正确还原。
- **V3**：抽查的跨文档链接可点击跳转到对应文档。
- **V4**：本地 13 个文件内容与对应飞书文档内容一致（迁移当下应逐一 `fetch` 比对一次，确认无转换损耗）。
- **V5**：本地 `README.md` 与 `plan/README.md` 的分发声明已更新，明确"飞书是主入口，本地是工作副本"。

## 7. 已知风险与不处理的点

- **格式转换损耗**：markdown → 飞书 docx 的转换是黑盒，可能存在个别语法不支持或渲染走样（比如深层嵌套列表、某些表格写法）。全量迁移意味着如果出现系统性问题，13 个文档要一起返工——这是用户在"试点 vs 全量"的选择中已经接受的风险。
- **锚点级跳转体验**：本版明确不做，跨章节引用只能跳到文档首页再自行查找，属于已知的体验降级，非 bug。
- **飞书版本历史弱于 git diff**：出现"手滑改坏"的情况，飞书自己的版本历史可以回滚，本地 git 作为第二道恢复手段。
