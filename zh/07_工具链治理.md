# 07 工具链治理：给 AI 造的轨道自己也会腐化

## 核心定义

Workflow、skill、审查命令、钩子脚本——这些为 AI 协作搭的"轨道"，本身也是代码：会有死代码、会有契约断线、会和文档脱节、会被 agent 绕过。

工具链治理回答四个问题：

- 轨道自身按什么工程标准维护？
- 一条约束应该由**谁**执法——文档、审查，还是机器？
- 无人值守时，怎么保证 agent 绕不过轨道？
- 使用者无法核查 AI 时，治理怎么整体升压？

## 1. Skill 也是代码

### 1.1 适用同一套工程纪律

| 代码的纪律 | 在 skill 上的形态 |
|---|---|
| 单一职责 | 一个 skill 管一段流程；诊断的不动手术，审查的不顺手修 |
| 死代码清理 | 永远走"不存在则跳过"分支的步骤，按治理债处理（见《技术债与经验回流》§4.1） |
| 变更要 review | 改 skill 机制 = 改流程契约，值得和改代码一样对待 |
| 文档同步 | skill 机制变更后，引用它的方法论文档欠一次同步——改了没同步 = 没改完 |

### 1.2 跨工具读写契约：两端一起设计

工具链最隐蔽的断线：工具 B 设计为读取工具 A 落盘的文件，而 A 的定义里从没有写盘步骤。B 每次都走"文件不存在 → 跳过"，机制从未生效，也从不报错（完整案例见《案例库》#23）。

规则：

- 读端定义时，写端必须同时写明：**由谁、在哪一步、写什么格式**。
- 读端注明写端出处，让断线可审计。
- 连续 N 次全走跳过分支的机制，按治理债清理。

## 2. 执法层级：约束由谁兜底

同一条约束，可以由五个层级执法。层级越高，agent 绕过的可能性越低，建设成本也越高：

| 层级 | 执法者 | agent 能否绕过 | 典型形态 |
|---|---|---|---|
| L1 模型自律 | agent 自己 | 随时（无意漂移即绕过） | 口头约定、对话提醒 |
| L2 文档规则 | 常驻规则文件 | 能（注意力稀释、规则冲突时） | AGENTS / CLAUDE 条目 |
| L3 review 执法 | 审查环节 | 较难（但审查本身可能被跳过） | plan / change review checklist |
| L4 机器闸门 | 脚本 / lint / CI | 很难（除非改闸门本身） | lint 规则、grep 闸门、测试 |
| L5 harness 硬拦 | 运行环境钩子 | **不能**（在 agent 够不着的层级） | pre-commit 拦截、权限钩子 |

两条推论：

- **规则升级的终点是升层级**，不是把文档写得更严厉。同类问题反复出现时，问的不是"要不要再强调一遍"，而是"能不能升到 L4/L5"。
- **越接近无人值守，要求的执法层级越高。** 有人盯着时 L2/L3 够用；无人时，关键约束必须在 L4 以上——L1/L2 靠的是 agent 自觉，而无人值守的前提恰恰是不依赖自觉。

常见误区：把"升级到机器闸门"等同于"需要先建 CI"。本地 pre-commit 钩子本身就是不依赖 CI 的 L4 闸门——复发计数触发升级时，先看能不能用最小成本的本地闸门立刻拦住，不要把升级动作无限期押注在"等 CI 搭好"上。

## 3. Verdict 物证化

无人值守流程的自动推进，凭据必须是**物证**，不是声称：

- 审查通过与否，只认落盘文件里的 verdict 行；agent 口头说"review 过了"无效——声称无法被事后证伪，物证可以。
- 硬闸门装在 L5：无人值守标记存在时，提交动作被运行环境钩子拦截，验证落盘 verdict == APPROVE 才放行。agent 哪怕漂移了，也提交不出去。
- 标记文件圈定生效范围：进入无人值守时创建、退出时删除——闸门不干扰有人值守时的正常裁量（如显式跳过 review）。
- 每次自动通过留一行审计记录，信任但可抽查。

同理，"验证做过了"也要物证化：检查点报告"已验证 / 未覆盖"一行（见《长任务、Loop、Cron 与无人值守》§5.6），而不是散落在过程叙述里。

## 4. 假绿防御是闸门的自证义务

L4 闸门自己也会腐化成假绿：扫描路径失效、规则空跑、测试数为 0，却仍然报 SUCCESS。所以闸门必须**自证执行**：

- 输出检查对象数量（扫描文件数 / 测试数 / 断言数），计数为 0 按失败处理。
- 定期故意造一个已知违例，确认闸门真的能抓到。
- 完整清单见《长任务、Loop、Cron 与无人值守》§5.7 假绿防御。

一种更隐蔽的假绿来源：**主动配置压制掉的警告**。构建工具的 warning 不会导致非零退出码，团队为了让输出"看起来干净"，可能通过配置项统一禁用某类告警——但告警里混杂着两种性质完全不同的东西：与项目其他明确决策冲突的噪声（可以安全 disable），和代表真实未完成工作的信号（一旦也被 disable，警告列表变干净了，但真实的技术债变得不可见）。压制告警前先问：这条 warning 是噪声，还是尚未完成的工作在提醒你？只压噪声，不压信号。

## 5. 方法论、模板与工具的三态关系

同一套机制往往存在三种形态，必须分清谁是权威：

| 形态 | 角色 | 权威范围 |
|---|---|---|
| 方法论文档（本知识库这类） | 讲**原理与设计理由**，自包含、可迁移 | why |
| 模板 | 可复制的骨架 | 起点 |
| skill / 脚本（落地实现） | **运行时准绳** | how 的最终裁决 |

规则：

- 机制落地成 skill 后，运行时以 skill 为准；方法论文档记录 why，与 skill 保持**机制级一致**（verdict 枚举、状态语义、落盘契约、闭环走向），措辞级差异不算脱节。
- skill 机制变更 → 方法论文档欠一次同步。同步纪律本身也要遵守执法层级：只写"要记得同步"是 L2，配一个哈希基线比对脚本才是 L4。

## 6. 非对称协作：为不能核查你的人设计 harness

前几节的执法层级默认使用者是工程师——看得懂 diff、能识别 AI 的错误。当使用者换成**无法核查技术正确性的角色**（产品、运营按需求文档直接驱动 AI 改代码），同一套治理要整体升压。核心判断：

> **守门强度与使用者的核查能力成反比。** 工程师自用，L2/L3 加人工判断够用；使用者无法识别错误时，五类失效模式（越界、漂移、脑补、自证完成、跨会话遗忘）必须全部由机制拦住，而不是指望 AI 自觉或对方的技术判断。

设计原则：

**先诊断缺哪层，再动工。** 一个真实标本：项目文档层治理已完备，执法层为空（无 hooks、无闸门）——"自证完成"与"review 漏检"的根因是执法缺层，不是规则不够。在已满的层上继续堆规则没有用，补缺的那一层（层级定义见 §2）。

**操作面最小化。** 使用者的全部操作收敛到极少数动作（贴需求、回答领域问题、看报告说过 / 不过），复杂度全部藏进机制。要求使用者跑命令、改文档、理解 verdict 文本的设计，都是把治理成本转嫁给了不该承担的人。

**语言隔离 + 决策路由。** 一切面向使用者的输出用对方的语言，零工程术语。问题按角色路由：只把对方能判断的问题（流程、文案、优先级、要不要忍某现象）给对方选；技术问题按项目宪法自决，宪法覆盖不了的显式标注"转工程师"——禁止上抛制造决策疲劳。故障出口统一且无脑（"把截图发给某人"）；环境故障显式化为独立状态，不吞掉、不硬撑。

**审查独立性靠冻结，不靠自律。** 实现者与审查者必须分离，且独立性不能依赖主 agent 派发时"客观转述"：

- 审查员的立场写死在版本管理的角色定义里（证伪导向：默认实现是错的，找不到证据才 PASS），主 agent 无法在派发时改写；
- **输入隔离**：派给审查员的只有需求编号与实现前冻结的验收锚点（使用者拍板过的验收步骤是天然独立基准），禁止附带实现描述、diff 摘要、"我已完成 X"式说明——实现者的自述是污染源；
- **只判不修**：审查员发现问题只记录，修复回到实现方；用权限执法（不给写权限），不靠纪律。裁判不下场踢球。

**物证链只保证"发生过"，不保证"结论对"。** 机器层能保证两件事：审核确实发生过（来源校验——用 harness 写入、agent 无法篡改的运行记录做证据）；验的就是交付的内容（物证与内容哈希绑定，不按时间配对，防"验的是 A、交的是 B"）。审核结论本身的正确性，终局锚点是**人看得懂的真实证据**（如验收报告里的实测截图）——好的防伪设计让伪造证据的成本 ≈ 把事情真做对的成本。

**逃生舱必须自动过期。** 任何旁路（"还在改先别拦"的 WIP 标记、防死循环的泄洪）都要带自动过期（如会话级清除）、有界计数、审计记录——不带过期的逃生舱必然演化成永久后门。两种角色共用一个仓库时，工程师侧旁路用机器可判定的本地开关（不入库、只存在于自己机器），而不是让闸门学会认人。

**Harness 自己要过验收。** 上线前对治理机制本身做端到端自验：闸门先对存量跑一遍基线（不冤枉存量）；端到端演练包含**对抗测试**——手工伪造一份通过物证，确认被拦下；模拟对方环境从零首跑一遍。这是 §1"skill 也是代码"的收尾：代码要测试，闸门要对抗测试。

**快闸门管每回合，慢巡检管漂移。** 每回合的强制检查只放高频失效模式；全量回归、机制与文档的漂移检查放进低频人工巡检——不要把所有防线塞进每回合钩子，故障面本身也是成本。

与既有章节的关系：拒手段不拒目的（《上层定法》§1.6、案例库 #28）在这类场景是硬要求——使用者被生硬拒绝后绕过 AI 手改文件，护栏全灭；立债权归人（《技术债与经验回流》§4）在这里升级为机器执法——无拍板记录的新增债务，审查直接判不通过。

## 7. 工具链治理检查清单

- [ ] 每个 skill 职责单一，没有顺手越界的步骤？
- [ ] 跨工具的读写契约两端齐全，读端注明写端出处？
- [ ] 有没有连续空转的流程环节（治理债候选）？
- [ ] 每条关键约束标注了执法层级？无人值守路径上的约束 ≥ L4？
- [ ] 自动推进的凭据是落盘物证，不是口头声称？
- [ ] 机器闸门有自证执行（计数 > 0）？
- [ ] 方法论 / 模板 / skill 三态的权威关系已声明？同步纪律有机械触发物？
- [ ] 使用者无法核查输出时，关键约束是否全部升到 L4/L5？审查独立性是否机制化（角色冻结 / 输入隔离 / 只判不修）？
- [ ] 逃生舱有自动过期与审计？harness 上线前做过对抗自验（伪造物证会被拦）？
