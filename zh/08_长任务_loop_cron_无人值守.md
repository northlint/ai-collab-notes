# 08 长任务、Loop、Cron 与无人值守

## 核心定义

当 AI 任务跨越长时间、多步骤、多外部系统、甚至需要自动续作时，普通“一次对话内完成”的工作流不够。

这时需要专门设计执行编排系统：

- cron：定时触发
- watch：轻量看守
- worker：串行执行器
- loop：长程自治循环
- queue：任务队列
- handoff：跨会话交接
- checkpoint：人工卡点
- oracle：自验证神谕

这部分解决的是：

> AI 如何在长任务中不断电、不乱跑、不猜补、不越权。

## 1. 长任务为什么不同

长任务有几个特点：

- 任务可能超过单次会话时长。
- 任务需要等待外部输入。
- 任务需要跨时间点继续。
- 任务可能依赖设备、浏览器、Figma、真机、测试环境。
- 任务中可能出现阻塞。
- 任务完成标准不是一次命令能判断。

如果没有执行编排，AI 容易：

- 中断后无法接手
- 重复做已经完成的部分
- 在缺工具时停摆
- 等待人类输入时丢上下文
- 无人值守时猜测推进
- 多任务同时跑导致分支和状态打架

## 2. Cron：定时触发与兜底

### 2.1 Cron 适合什么

cron 适合：

- 定时启动下一阶段
- 兜底复查
- 周期性巡检
- 固定频率拉取任务
- 会话可能断掉时的续作

典型场景：

```text
21:10 启动 B2
如果前序未完成，先收尾前序
02:10 再兜底检查是否全部完成
若已停在检查点，不重复执行
```

### 2.2 Cron 不适合什么

cron 不适合：

- 高频实时消息响应
- 需要立即处理的交互
- 强依赖当前会话上下文的任务
- 多任务并发写同一仓库

cron 的问题：

- 粒度粗
- 容易重复触发
- 上下文弱
- 必须有幂等和锁
- 必须知道“当前状态是否已经完成”

### 2.3 Cron 续作必须有 handoff

定时续作必须提供 handoff 文档。

handoff 至少包含：

- 用户目标原话
- 当前阶段
- 已完成清单
- 未完成清单
- 真相源
- 工作流
- 已定决策，不得推翻
- 正常停止点
- 如果工具缺失如何降级
- 哪些状态是“符合预期，不要重复做”

没有 handoff 的 cron，就是让新会话盲开任务。

### 2.4 轮数未知时：自举式一次性触发链

跨窗口续跑的任务常常说不准"要跑几轮"。预先写死 N 次触发是反模式：N 小了任务没跑完且无人发现（静默失败），N 大了空转浪费。

收敛的设计是让触发链**自举**：

```text
第 1 次触发（人工创建，一次性）
  ↓ 执行时先按 durable spec 做完成判定
  ├─ 已完成 → 收尾总结，结束，不再挂新触发
  └─ 未完成 → 跑一次工作增量 → 再判定
       ├─ 已完成 → 收尾，结束
       └─ 仍未完成 → 创建下一个一次性触发（prompt 与本次相同，自我复制）
```

需要几轮就跑几轮，做完自动停。两个前提：

- 完成判定必须**具体到可脚本化检查**（存在 N 个非空文件 / 某命令输出为 0 / 测试全绿），模糊目标排出去的触发链会在无人时跑偏；
- 触发链若是**会话内存态**（进程一退全部无声消失、不报错），这个脆弱性假设必须在启动前向用户复述确认，不要"当作能行"。

## 3. Watch：当前窗口模式的轻量看守

### 3.1 Watch 是什么

watch 是后台轻量进程。

它不执行完整任务，只负责观察外部事件。

发现事件后：

```text
打印事件摘要
退出进程
唤醒主会话
由主会话继续处理
```

### 3.2 Watch 适合什么

watch 适合：

- 当前窗口有人看着
- 需要保留主会话上下文
- 外部事件不频繁
- 执行需要正常权限和上下文
- 不适合嵌套无头 agent

典型场景：

```text
每 10 秒查一次：
  - 是否有新 pending bug
  - 群里是否有人 @ 指定机器人
发现任一：
  - 输出 reason / bug / message
  - process.exit(0)
  - 主会话接手
```

### 3.3 Watch 的设计要点

- 只观察，不修改代码。
- 只发信号，不做复杂决策。
- 用 marker 去重。
- 外部 API 偶发失败，下轮重试。
- 发现事件后退出，而不是自己处理。
- 主会话处理期间 watch 停止，避免并发。

### 3.4 Watch vs Cron

| 维度 | Watch | Cron |
|---|---|---|
| 触发方式 | 持续观察，发现即退出 | 定时启动 |
| 上下文 | 保留当前会话 | 新会话或新进程弱上下文 |
| 适合 | 当前窗口模式、等人回复、轻量队列 | 定时兜底、周期任务 |
| 风险 | 会话关闭后失效 | 需要幂等和锁 |

## 4. Worker：串行完整执行器

### 4.1 Worker 是什么

worker 是能处理一条完整任务链路的执行器。

比如自动修 bug：

```text
拉 pending bug
  → 占用任务
  → 诊断
  → 发群
  → 等人工意见
  → 拉 fix 分支
  → 修复
  → change-review
  → commit
  → 通知
  → 复位
```

### 4.2 Worker 的关键设计

worker 必须有：

- 串行锁
- 任务占用
- 状态回写
- 超时处理
- 错误通知
- 分支隔离
- 不 push / 不合并的安全边界
- 人工卡点
- 失败后复位

### 4.3 Worker 的风险

worker 比 watch 风险高，因为它会真实执行任务。

风险包括：

- 嵌套 agent 权限过大
- 无上下文导致误改
- 等人工输入时占锁太久
- commit 前格式化钩子改动未核
- 外部 API 超时导致半流程
- 分支未复位

所以 worker 初期应该：

- 只处理一个任务
- 不 push
- 不合主干
- commit 后人工 review
- 全程记录状态

## 5. Loop：长程自治循环

Loop 的对象不是 bare task，而是 workflow。

如果只是让 AI：

```text
做任务 → review → 修 → review → 再修
```

它很容易陷入 review 无穷债。

因为 review 是一条没有天然零点的梯度：风格、边界、抽象、命名、可维护性永远还能继续挑。

真正能收敛的 loop，必须把 done 从“reviewer 没话说了”改成：

> 对齐固定靶子 + 通过机器门禁 + 动态自检通过 + review 只对固定靶子做 verdict。

### 5.1 Loop 和 worker 的区别

worker 处理一条明确任务。

loop 是长期运行的自治系统。

loop 需要处理：

- 长程心跳
- 阻塞检测
- 任务重排
- 设备看门狗
- 连续空跑计数
- 审核队列
- 部分自动 ship
- 模糊项 park

### 5.2 Loop 的典型结构

```text
loop
  → 读取任务队列 / 状态
  → 选择下一个可执行任务
  → 执行一个最小 pipeline
  → 跑机械门禁
  → 跑 oracle 验证
  → 有把握则交付
  → 不确定则 park 到审核队列
  → 更新状态
  → 下一轮
```

### 5.3 Loop 的终止条件

loop 不能无限跑。

需要明确终止条件，例如：

- 连续 N 轮无新增任务
- 所有任务都完成或 park
- 关键依赖不可用
- 设备不可恢复
- 人工审核队列超阈值
- 触及高风险操作

示例：

```text
连续 3 轮检测无新增需处理逻辑
  → 视为收敛
  → 停 loop
  → 发通知报告结果
```

### 5.4 Loop 的安全阀

loop 必须有安全阀：

- park-don't-guess：不确定就 park，不猜。
- 人工审核队列：模糊项进入队列。
- 设备看门狗：设备卡死重启或重排。
- 门禁失败停止：构建/测试不过不继续堆改动。
- 任务粒度小：每轮只推进一个最小闭环。

### 5.5 闭环收敛五要件

一个 loop 能收敛，至少需要五件事。

| 要件 | 作用 | 例子 |
|---|---|---|
| 固定靶子 oracle | 给 done 一个外部定义 | spec、openspec、ux、活实现、状态矩阵 |
| 两条独立度量 | 同时抓静态形状和动态行为 | lint / grep / schema + Playwright / adb / 真机 |
| 学习棘轮 | 每轮要么修缺陷，要么锁死一类复发问题 | lessons、AGENTS、skill、CI |
| 无人时保守动作 | 无人盯时不做不可逆或高歧义动作 | park、登记 TD、等待人工 |
| 可信传感 + 独立验证 | 防止自我确认和假验证 | 校准浏览器/设备 + 对抗审查 |

缺固定靶子，loop 会追移动目标。

缺两条度量，loop 会只证明“代码形状对”或“看起来能跑”。

缺学习棘轮，loop 每轮都在重复踩坑。

缺保守动作，loop 会在无人时越权。

缺可信传感，loop 会拿错误截图、旧缓存或空跑门禁给自己盖章。

### 5.6 静态门禁与动态自检必须配对

静态门禁回答：

- 代码形状是否合规
- 架构边界是否被破坏
- 禁用模式是否出现
- 类型、lint、测试是否通过

动态自检回答：

- 运行时是否正确
- 页面是否真实渲染
- 真机交互是否可用
- 异步和错误态是否符合预期

两者不能互相替代。

静态全绿，不代表跑起来对。

动态截图正常，也不代表没破坏架构边界。

动态自检的时机同样重要：**边做边验，不留事后补**。有可运行界面的改动，应在实现过程中即时打开真实页面走关键路径——事后集中补验，问题的修复成本已经翻倍。并且每个检查点必须报告一行：

```text
已验证：<真实跑过的路径>
未覆盖：<没跑到的路径>
```

这一行把"验证覆盖面"变成审查可见的物证：审查者不用重复怀疑已验证的路径，也能一眼看到盲区。无法运行验证时如实标"未运行验证"，不默认通过。

### 5.7 假绿防御

无人 loop 里，假绿比红灯更危险。

假绿指的是系统显示通过，但实际没有验证到目标。

典型假绿：

- 门禁脚本路径错，扫描目录不存在，但仍 SUCCESS。
- grep 过滤器漏改，规则空跑。
- 测试命令执行了，但测试数量为 0。
- 浏览器打开了 `file://` 或旧缓存页面。
- Playwright 视口不是目标宽度，却按目标宽度验收。
- 设备跑的是旧包，截图来自旧代码。
- UI 验证只断言 DOM 文本：文案、金额逐项全对，布局已断（内容挤压换行、图标拉伸变形），文本断言全部通过（案例库 #29）。

防御方式：

- 检查执行计数必须大于 0。
- 故意造一个已知违例，确认门禁能抓到。
- 禁止用 `file://` 作为最终动态验收。
- 使用 cache-bust 或清缓存。
- 记录真实视口、设备、包版本、构建号。
- 对关键 oracle 做独立复核。
- **断言维度覆盖 bug 可能出现的维度**：文本断言只覆盖内容维度；视觉改动必须真看截图收尾，必要时加几何断言（元素坐标判同行、宽高比判变形）。

### 5.8 传感器校准

传感器指任何用来判断结果的工具：

- 浏览器截图
- 真机截图
- 控制台日志
- 网络请求记录
- 测试报告
- lint 输出
- 构建产物
- API fixture

传感器使用前要先校准。

校准要问：

- 它看到的是不是当前代码？
- 它看到的是不是目标环境？
- 它有没有缓存？
- 它的分辨率、设备、账号、数据是否正确？
- 它失败时会不会被吞掉？
- 它空跑时会不会仍然成功？

一句话：

> 没校准的验证，比没有验证更危险。

## 6. 无人值守作业

### 6.1 无人值守不是没人负责

无人值守不是“让 AI 自由跑”。

它是：

> 在强约束、强验证、可回滚、有人类审核队列的前提下，让 AI 自主推进确定性任务。

### 6.2 无人值守成立条件

无人值守必须具备三类东西。

#### 1. 冻结真相源

任务目标不能是移动靶。

例如：

- 业务能力已冻结
- H5 已 feature-complete
- Spec 已稳定
- 状态矩阵已清楚

如果真相源还在变，无人值守会追移动靶。

#### 2. 对错 oracle

AI 不能只自证完成。

需要外部神谕：

- 活的参照实现
- 真实后端
- mock 状态矩阵
- 截图对比
- 真机行为
- 测试环境
- 端到端验收脚本

没有 oracle，无人值守输出只能全审。

#### 3. 机械门禁

必须有自动阻断：

- build
- lint
- test
- typecheck
- invariant check
- grep
- screenshot
- console
- API fixture

机械门禁还必须证明自己真的执行了——自证执行的完整清单（检查计数 > 0、记录真实环境、失败非零退出、空跑判失败）见 §5.7 假绿防御，执法归属见《工具链治理》§4。

#### 4. 人工卡点与保守动作

无人值守不是绕过人类判断。

以下事项必须保守处理：

- 业务规则冲突
- 真相源冲突
- 删除或覆盖数据
- 修改公开契约
- push / merge / release
- 无法自动判断的视觉差异
- 需要产品或架构拍板的方案分歧

无人时的默认动作：

- 不做不可逆操作。
- 不反转已记录决策。
- 不扩大任务边界。
- 模糊项 park 到审核队列。
- 缺口登记技术债。
- 在 handoff 中列出醒后复验清单。

### 6.3 无人值守适合什么任务

适合：

- 大量重复迁移
- 已冻结能力的跨端重写
- 状态矩阵明确的 UI 补齐
- DTO / schema / contract 镜像
- 明确 bug 修复链路中的低风险部分

不适合：

- 产品决策
- 架构方向选择
- 契约不明
- 视觉无源
- 高风险写操作
- 需要大量主观判断的交互

## 7. 事件订阅、轮询与定时

### 7.1 三种触发方式

| 方式 | 适合 | 注意 |
|---|---|---|
| 事件订阅 | 实时消息、Webhook、系统事件 | 需要权限和服务稳定 |
| 轮询 | CLI 可读但无订阅、MVP 快速验证 | 要处理超时、去重、频率 |
| 定时 cron | 周期巡检、续作、兜底 | 要幂等、锁、状态判断 |

### 7.2 选择原则

- 能稳定订阅的实时事件，用事件订阅。
- 权限成本高或 MVP 阶段，用轮询。
- 明确时间点或兜底检查，用 cron。
- 当前会话有人看着，用 watch。
- 要完整自动执行，用 worker。
- 要持续自治，用 loop。

## 8. 人工卡点

### 8.1 哪些必须卡点

必须人工卡点：

- 合并生产分支
- push 到共享远端
- 删除资源
- 改架构不变量
- 接受高风险技术债
- 契约冲突裁决
- 产品体验取舍
- 无法被 oracle 验证的结论

### 8.2 卡点应该怎么设计

卡点不能是口头“等你看看”。

要提供：

- diff
- 分支
- commit
- 验证结果
- 截图
- 风险说明
- 可选动作
- 明确批准方式

例如：

```text
修复已提交到本地分支，不 push。
请人工 review 后决定：
1. 继续 push
2. 退回修改
3. 放弃该分支
```

### 8.3 检查点协议：把卡点从习惯变成协议

有人值守时的卡点，同样需要协议化，否则会被"顺手推进"侵蚀。一个经过实践的检查点协议：

- **产出后必停**：产出计划 / 完成改动后必须暂停，显式提供选项（review（推荐）/ 跳过直接下一步 / 讨论），不自动推进。
- **verdict 驱动流转**：review 通过 → 等确认再下一步；有 must-fix → 修完提供增量 re-review；根本性问题 → 修完必须重新 review，无跳过。
- **增量 re-review**：只核对上轮问题是否修复，不重跑全量——否则 re-review 成本会让人倾向跳过它。增量的前提是上轮 must-fix 清单**还在手上**（落盘物证，或仍在当前上下文）；清单已随会话丢失 / 被压缩时必须走全量，凭印象做增量必然漏。

无人值守时，检查点协议**降级为 verdict 确认**，规则必须写死：

- **唯一自动分支**：落盘 verdict == APPROVE → 自动继续；**其余任何 verdict 一律停下等人**。
- 措辞上不留"问题不大可继续"这类模糊地带——agent 的自我豁免全部发生在规则的模糊地带里，规则无模糊地带就豁免不了。
- verdict 只认**落盘物证**，口头声称无效——物证化与审计的完整要求见《工具链治理》§3（那里是主场）。

## 9. Git 写操作三档模式

自动化系统必须支持降级。

| 模式 | push | 合并 | 适用阶段 |
|---|---|---|---|
| C 全人工 | 机器只给分支、diff、命令 | 人工合并 | MVP / 高风险 |
| B 半自动 | 机器 push 特性分支 | 人工 MR / merge | 跑稳后 |
| A 有批准的全自动 | 机器 push | 人工批准后机器合并 | 高成熟度 |

默认从 C 开始。

不要一开始就追求全自动。

## 10. Handoff：跨会话交接

长任务必须能被新会话接手。

handoff 文档应包含：

- 任务目标
- 当前状态
- 已完成事项
- 未完成事项
- 不得推翻的决策
- 真相源
- 工作流
- 验收方式
- 正常停止点
- 异常处理方式
- 如果工具不可用如何降级

一个重要规则：

> 停在检查点也算完成某阶段职责，不要让续作会话重复构建。

## 11. Park-don't-guess

无人值守和长任务里，最重要的安全原则是：

> 不确定就 park，不靠猜补。

park 的意思是：

- 记录问题
- 写明需要人判断什么
- 保留当前产物
- 不继续扩大改动
- 放入审核队列或技术债

常见 park 场景：

- 源冲突
- 视觉无源
- 契约不明
- 真账号造不出状态
- 设备异常
- oracle 无法判断
- 高风险操作需批准

## 12. 长任务设计检查清单

设计长任务前，检查：

- 任务是否需要跨会话？
- 是否有 handoff？
- 是否有状态记录？
- 是否需要 watch、cron、worker 还是 loop？
- 是否有锁防并发？
- 是否有去重 marker？
- 是否有人工卡点？
- 是否有失败恢复？
- 是否有 oracle？
- 是否有 park 队列？
- 是否有终止条件？
- 是否有通知机制？

如果这些都没有，不要叫无人值守。
