# 附录 A：案例库

## 使用方式

案例库不是故事集，而是训练判断力的材料。

每个案例都按三段写：

- 反模式：AI 或团队容易怎么做错。
- 正解：应该如何纠偏。
- 抽象规则：沉淀成可复用的治理原则。

案例可以被用于：

- onboarding
- review checklist
- spec 模板示例
- 自动化工作流设计
- 子 agent 任务说明
- 技术债复盘

## 1. 把视觉源误当业务源

### 反模式

AI 根据设计稿里的文字、按钮顺序、状态标签，直接推断业务规则。

例如：

- 设计稿里有“审核中”，AI 就假设接口状态字段一定叫 `pending`。
- Figma 页面没有展示失败态，AI 就认为失败态不存在。
- 视觉稿里列表只有三条数据，AI 就把分页逻辑省掉。

### 正解

视觉源只回答“长什么样”。

业务规则必须从业务 spec、生产行为、接口事实、已验证逻辑中确定。

如果视觉源和业务源冲突：

- 先标记冲突
- 明确真相源优先级
- 必要时请求人工裁决
- 裁决后回写 spec

### 抽象规则

> 设计稿是视觉真相源，不是业务真相源。

## 2. 接口文档与生产事实冲突

### 反模式

AI 只读取旧接口文档，按文档字段实现。

但生产接口实际已经改字段，旧 mock 和旧测试又刚好支持旧字段，导致测试通过但线上失败。

### 正解

接口类任务必须区分：

- 历史文档
- 当前 openspec
- 真实请求样例
- 生产观测行为
- 现有代码调用事实

当这些来源冲突时，不能直接选择最容易读取的文档。

应按项目定义的真相源优先级裁决，并把冲突登记为文档债。

### 抽象规则

> 接口实现以当前事实为准，旧文档只能作为线索。

## 3. AI 声称测试通过，但没有断言

### 反模式

AI 增加测试文件，运行命令通过，然后声明完成。

审查发现：

- 测试只渲染组件
- 没有断言关键行为
- 没有覆盖异常路径
- mock 数据和真实风险无关
- 测试名字面上暗示了某个场景（如 `invokesTracker` / `logsWarn`），但 body 实际断言的是另一件事——要对照测试名字面含义和 body 真实断言内容是否一致，不能只看测试是否存在

### 正解

测试必须证明任务目标。

至少要问：

- 测试断言了什么？
- 如果实现是错的，测试会失败吗？
- 是否覆盖了本次 bug 的复现路径？
- 是否覆盖边界状态？

### 抽象规则

> 测试通过不等于验证完成；没有断言的测试接近摆设。

## 4. 异步失败被静默吞掉

### 反模式

长任务里某一步 API 调用失败，catch 之后只打印日志。

主流程继续执行，最后输出“全部完成”。

### 正解

先定义错误等级：

- 阻断错误：停止流程，写入失败状态，通知人类。
- 可恢复错误：重试，超过上限后降级。
- 非阻断错误：记录 warning，不影响主流程。

自动流程结束前必须检查失败列表。

### 抽象规则

> 无人值守流程里，静默失败就是伪完成。

## 5. 技术债没有登记

### 反模式

AI 发现旧文档过期、某状态缺测试、某 worker 无复位，但因为本次任务还能完成，就只在最终回答里提一句。

下一次任务换了上下文，问题再次发生。

### 正解

凡是满足以下条件的问题，应登记技术债：

- 当前不修
- 后续会影响判断
- 有复发风险
- 有明确影响范围

登记时写清楚：

- 现象
- 风险
- 暂缓原因
- 偿还方式
- 验收标准

### 抽象规则

> 不登记的债务，会变成下一次 AI 的盲区。

## 6. 子 agent 空跑

### 反模式

主 agent 分配审查任务给子 agent。

子 agent 输出“未发现问题”，但报告里没有文件引用、没有命令证据、没有检查范围。

### 正解

子 agent 的产出必须可审计。

主 agent 应要求：

- 明确读取了哪些文件
- 明确审查维度
- 明确没有覆盖哪些范围
- 发现项带路径和证据

关键任务中，主 agent 必须抽查。

### 抽象规则

> 子 agent 能分担工作，不能分担最终责任。

## 7. Cron 续作导致重复工作

### 反模式

凌晨 cron 启动新会话，新会话不知道前一阶段已经完成，于是重复执行批任务。

结果：

- 覆盖前一次改动
- 重复提交
- 状态混乱
- 人类难以判断哪个结果可信

### 正解

cron 续作必须有 handoff。

handoff 要说明：

- 已完成什么
- 未完成什么
- 当前停在哪个检查点
- 哪些状态是正常停留
- 哪些动作不能重复
- 如何判断是否继续

cron 任务本身必须幂等。

### 抽象规则

> 没有 handoff 的 cron，是盲开新任务。

## 8. Watch 与 Worker 混用

### 反模式

本来只需要监听群消息，却让后台 worker 直接修代码、commit、通知。

由于 worker 无上下文，误把临时讨论当成正式任务。

### 正解

区分 watch 和 worker：

- watch 只观察事件，发现后退出，唤醒当前会话。
- worker 处理完整任务链，必须有锁、状态机、人工卡点。

当前窗口有人值守时，优先 watch。

真正无人值守且流程成熟时，才使用 worker。

### 抽象规则

> 能用 watch 保留上下文时，不急着上 worker。

## 9. 无头 worker 被权限和上下文卡住

### 反模式

为了自动化，把完整修复链交给无头 agent。

结果无头 agent：

- 没有当前会话上下文
- 工具权限不一致
- 等人工输入时卡死
- 错误处理不完整

### 正解

先采用 current-session 模式：

- 后台进程只监听
- 事件到达后退出
- 当前主会话继续处理

等工作流稳定后，再把明确、低风险、可状态化的部分下沉到 worker。

### 抽象规则

> 自动化成熟度不足时，先自动唤醒，不要自动决策。

## 10. 无人值守迁移没有 oracle

### 反模式

AI 批量迁移页面，运行构建通过后就认为完成。

但迁移目标其实包括：

- 视觉一致
- 业务状态一致
- 接口字段一致
- 真机交互一致
- 异常路径一致

构建通过无法证明这些。

### 正解

无人值守迁移必须提前定义 oracle。

oracle 可以包括：

- 源页面实时对照
- 状态矩阵
- schema 校验
- screenshot diff
- 真机操作脚本
- 控制台错误检查
- 网络请求断言

没有 oracle 的任务，不能承诺无人值守完成。

### 抽象规则

> 长任务能不能无人值守，取决于有没有可自动判断的完成标准。

（迁移类任务的 oracle 设计与准入前提，见 [`09_迁移与重写治理.md`](./09_迁移与重写治理.md) §5。）

## 11. Mock 被当成假数据

### 反模式

AI 把 mock 仅当成本地调试数据，随意改成能让页面好看的样子。

结果 mock 不再覆盖真实业务状态。

### 正解

在迁移、重构、状态机任务中，mock 应作为状态台账。

它应该覆盖：

- 成功
- 空态
- 加载
- 错误
- 边界值
- 权限差异
- 多状态组合

mock 改动必须说明对应业务状态。

### 抽象规则

> 高质量 mock 是状态 ledger，不是假数据。

## 12. “看起来正常”不等于设计还原

### 反模式

AI 打开页面，看一眼没有明显错位，就说 UI 完成。

但实际存在：

- 字体大小不一致
- 间距偏差
- 状态缺失
- 移动端换行错误
- 滚动区域不一致
- 图标资源错误

### 正解

设计还原需要明确验收方式。

常见方式：

- 与设计稿或源页面并排对照
- screenshot diff
- 多视口检查
- 关键组件状态截图
- 控制台错误检查
- 真机验证

### 抽象规则

> UI “能渲染”不是“还原完成”。

## 13. 缺资源后自由发挥

### 反模式

AI 找不到图标或图片，直接用占位图、渐变、纯色块替代，并把页面标记完成。

### 正解

资源缺失时应按顺序处理：

1. 搜索当前项目资源目录。
2. 搜索上游项目或旧版本。
3. 检查设计稿导出资源。
4. 询问或登记资产债。
5. 使用占位时明确标记不可最终验收。

### 抽象规则

> 缺资产不是自由创作授权，先找源，再占位。

## 14. 合约变更没有独立验证

### 反模式

AI 修改 schema、接口调用、UI 字段展示，但没有独立验证变更来源。

最后代码、mock、文档一起错。

### 正解

合约变更至少需要两类证据：

- 来源证据：字段来自哪里。
- 行为证据：实现如何被验证。

例如：

- openspec 示例
- 实际 API 响应
- 旧代码调用链
- 后端确认
- schema 测试

### 抽象规则

> 合约变更不能自证，必须有独立来源。

## 15. 批任务没有 Phase 0

### 反模式

多个页面、多个模块、多个 agent 同时开工。

每个 agent 自己建 mock、自己定义类型、自己处理缓存。

最后集成时发现：

- 类型重复
- 缓存失效策略冲突
- 目录结构不一致
- 组件边界不一致

### 正解

批任务要先做 Phase 0：

- 公共类型
- 共享 mock
- 数据契约
- 基础组件
- cache/invalidate 约定
- 验收标准
- 分工边界

然后再进入 Phase 1 并行实现。

### 抽象规则

> 并行之前先统一地基。

## 16. 人工卡点被省略

### 反模式

自动修复流程直接从诊断进入修改、commit、通知。

但诊断阶段存在业务判断，需要人类确认。

### 正解

把不可自动裁决的步骤设为人工卡点。

常见卡点：

- 需求歧义
- 业务规则冲突
- 删除数据
- 改公开接口
- push / merge / release
- 大范围重构

卡点消息要明确：

- 当前判断
- 需要人类选择什么
- 默认不做什么
- 超时如何处理

### 抽象规则

> 自动化不能越过业务裁决权。

## 17. Park-don't-guess

### 反模式

AI 缺少关键来源，但为了继续推进，猜一个最可能的方案。

猜测之后继续写代码，导致后续大量改动建立在不确定基础上。

### 正解

遇到关键不确定，应停在可恢复位置。

动作包括：

- 记录已知事实
- 记录缺失信息
- 说明不能推进的原因
- 给出下一步需要的输入
- 不做不可逆改动

### 抽象规则

> 缺关键真相源时，停住比猜对更工程化。

## 18. Commit 前没有核对实际 diff

### 反模式

AI 完成修改后直接 commit。

但格式化、生成文件、自动修复脚本引入额外改动，AI 没有检查。

### 正解

commit 前至少检查：

- `git status`
- 实际 diff
- 是否包含无关文件
- 是否有生成物或缓存
- 是否遗漏文档
- 是否通过必要验证

高风险任务还要做 change-review。

### 抽象规则

> commit 是交付动作，不是保存动作。

## 19. Review 报告没有 verdict

### 反模式

审查报告写了很多建议，但没有明确能不能继续。

执行 agent 不知道该修、该停、还是可以交付。

### 正解

review 必须给 verdict：

- `APPROVE`
- `APPROVE WITH CHANGES`
- `RETHINK`
- `REQUEST CHANGES`

并把问题分为：

- Blocking
- Important
- Minor

### 抽象规则

> 没有 verdict 的审查，不能驱动流程。

## 20. 经验只写在最终回复里

### 反模式

AI 在最终回答中总结“以后要注意”，但没有写入任何持久位置。

下一次任务完全无法继承。

### 正解

有复用价值的经验应回流到：

- 知识库
- AGENTS.md
- 模板库
- review checklist
- 技术债台账
- 自动化脚本

### 抽象规则

> 没有落盘的经验，不是系统能力。

## 21. 治理机制也会死亡

### 反模式

为"计划审查的 must-fix 被执行期遗忘"设计跨阶段账本：plan review 把问题落盘，change review 逐项核对关闭。机制精巧完整，运行后却发现账本永远为空——因为正确的流程本来就是执行前把 must-fix 直接修进计划、re-review 到通过。机制成了每次审查都要空跑一遍的死代码。

### 正解

废弃账本，源头修复：must-fix → 直接修订计划文档 → 增量 re-review → APPROVE 后才执行。同时把"账本之死"记录下来，防止后人重建。治理债的定义与判断标准见 [`06_技术债与经验回流.md`](./06_技术债与经验回流.md) §4.1。

### 抽象规则

> 治理机制防的问题若能在更源头零成本消掉，机制本身就是治理债。

## 22. 口头声称的 review 通过

### 反模式

无人值守 loop 中，agent 声称"review 已通过"，随即 commit 推进。实际 review 根本没跑，或跑了但 verdict 不是通过——声称无法被事后证伪，错误产物持续进入主干。

### 正解

verdict 物证化 + 分层执法：

- 自动推进的**唯一凭据**是审查落盘文件里的 verdict 行，口头声称无效。
- 在 agent 够不着的层级（运行环境的 pre-commit 拦截）加硬闸门：无人值守标记存在时，commit 必须验证落盘 verdict == APPROVE，否则直接阻断。
- 进入无人值守建标记文件、退出删除，让闸门只在无人时生效，不干扰有人值守的正常跳过。

### 抽象规则

> 口头声称不是物证；越接近无人值守，约束的执法层级必须越高。

## 23. 跨工具的读写契约断线

### 反模式

审查工具 B 设计为"读取工具 A 落盘的文件做闭环验证"，但工具 A 的定义里从头到尾没有写盘步骤。B 每次运行都命中"文件不存在 → 跳过"，这条精心设计的闭环**从未真正生效过**，且没有任何报错暴露它。

### 正解

- 跨工具的读写契约必须**两端一起设计**：读端定义时，写端必须同时写明"由谁、在哪一步、写什么格式"。
- 在读端注明写端出处（"此文件由 X 的第 N 步产生"），让断线可被审计。
- 定期对"条件跳过"分支做审计：连续 N 次全走跳过分支的机制，按治理债处理。

### 抽象规则

> 跨工具的闭环，先证明写端存在，再设计读端。

## 24. 组件抽取只做了一半

### 反模式

把重复出现的视觉样式下沉到共享样式层（比如把某个交互组件的 CSS 抽成公共类），但配套的行为代码（如自动跳格、状态管理）仍然留在每个调用页面里各自重复实现。表面上"已经 DRY 了"，实际上真正容易出错、维护成本最高的部分——行为逻辑——完全没有被抽取。

### 正解

抽取组件/逻辑时，把"视觉 + 行为 + 伴生元素"当作一个不可分割的整体单元一起下沉，不要只抽视觉层就宣布完成。判断是否抽完整很简单：如果同样的行为代码还在 ≥2 个调用方里各自出现一份，抽取就还没做完。

### 抽象规则

> 重复代码的抽取粒度是"整个功能单元"，不是"最容易抽的那一层"。


## 25. 知识库入口是文件清单，等于要求全读

### 反模式

项目建了定位用的事实库，但入口 README 写成文件清单："本目录包含 architecture / code-map / domain-glossary / user-journeys 四份文档。"agent 拿到 bug 后老老实实把四份全读一遍再动手——知识库不但没降低定位成本，反而变成每次任务的固定阅读税，和"没有知识库、直接全仓扫描"半斤八两。

### 正解

入口做成分诊路由器：第一刀问"换个页面会不会也这样"分横切 / 单点；横切走症状 → 机制表直达层，单点按描述给的抓手（界面原文 / 操作 / 名词 / 路由）由省到贵直达文件；并在入口显式写反指令——"不要把文档全读一遍再动手"。写进入口的每条定位路径必须用真实案例跑通过。

### 抽象规则

> 知识库入口的产品是路由，不是目录；不能分诊的索引等于要求全读。

## 26. 局部机制被写成全局机制

### 反模式

摸查时在某个页面发现一套换算 / 口径逻辑，直接写成"本系统的金额换算机制"。下次 bug 定位时，agent 按"全局机制"去修公共层，或把该口径套用到其他页面——修错了层，还波及无辜。

### 正解

断言机制范围前先横向 grep 同类页面：确认是"全局一处"还是"每页各自实现的模式"。症状 → 机制表加"范围"列，两种范围走不同修法：全局一处直接去改；每页模式只说明"该找什么形状的代码"，具体文件还要配合单点轴定页。

### 抽象规则

> 机制在哪些范围成立，和机制是什么同样重要；没核实范围的事实只算半个事实。

## 27. 「确认不存在」没有落盘

### 反模式

排查确认某功能不在本仓库（搜遍关键词零命中），只在对话里说了一句就过去了。下个会话、下个人遇到同类工单，把同样的关键词再搜一遍；更糟的是看到残留资源（死 i18n key、旧配置）误以为功能存在，在错误的地方越挖越深。

### 正解

事实库设"已排查确认不存在的功能"段：记录结论 + 排查依据（搜过哪些关键词、看过哪些目录），并标注校核日期——"不存在"也会过期。

### 抽象规则

> 确认不存在与确认存在是同等级的事实；不落盘的排查会被反复重买。

## 28. 护栏只会说"不"，人就会绕过它

### 反模式

两种对称的失败：其一，人（产品 / 业务）要求"直接把颜色改一下""手改生成物快一点"，AI 顺从执行——不变量被"经 AI 的改动"击穿；其二，AI 生硬拒绝、不给出路——对方嫌流程慢，绕过 AI 手动改文件，而护栏对"不经 AI 的改动"本就无力，从此形同虚设。

### 正解

拒手段，不拒目的：拒绝违规改法的同时，主动给出走正规链路达成同样效果的方案（改哪个源、几步、现在就可以做），把人导回安全通道；再配一道最小机械闸门（如 pre-commit 检查）兜住绕过 AI 的改动——AI 执法管说服，机械执法管兜底。

### 抽象规则

> 拒绝的产品是替代路径，不是"不"；只会说不的护栏，结局是被绕过。

## 29. 文本全对，视觉全错

### 反模式

UI 改动后，实现者用脚本断言页面文本——金额、文案逐项核对全部正确，随即报告"已验证"。人打开页面一眼看出两处问题：一行内容被挤压换行、图标被拉伸变形。文本断言碰不到布局与几何，验证维度与 bug 可能出现的维度错位——等于没验。

### 正解

- 断言维度必须覆盖改动可能破坏的维度：内容（文本）、布局（坐标、同行判断）、几何（宽高比）、渲染（真实截图）。
- UI 验证收尾必须真看截图；检查点报告的"已验证"一行，只有看过渲染结果后才允许写。
- 一个高频陷阱：矢量图标源文件声明了"不保持比例拉伸"（如 SVG 的 `preserveAspectRatio="none"`）时，容器比例与原始比例不符就会静默变形——文本层完全无感。

### 抽象规则

> 验证的维度必须覆盖 bug 可能出现的维度；文本断言过了 ≠ 页面对了。

## 30. 把旧注释里的"做不了"当免检通行证

### 反模式

代码注释 / 旧台账写着"此功能卡外部依赖，本仓做不了"。AI 每次路过都照抄这个结论按兵不动。实际上：依赖的挂钩点早已在契约表登记、同仓其他模块已用同一依赖跑通、项目约定里早有默认值模式——"做不了"在写下的那天也许成立，之后从没人重查。同一会话里连续两次把"待办"想当然归为"卡外部依赖"，都被人当场纠正。

### 正解

- 封锁性结论（"做不了 / 被阻塞 / 等第三方"）是最容易过期的负知识，采信前先证伪：查契约登记表 → 查同仓先例 → 查现成模式；三步查实仍阻塞才归为阻塞。
- 查到挂钩点就直接做，不停下来问"能不能做"。
- 对成熟参照系的绿地重写尤其如此：凡参照系统生产环境已跑通的能力，本仓大概率已有对应挂钩点——"没接线"≠"做不了"。

### 抽象规则

> 负知识要带日期使用；封锁性结论先证伪再采信，不是免检通行证。
