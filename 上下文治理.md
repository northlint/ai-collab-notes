# 上下文治理：把注意力当作稀缺资源

## 核心定义

上层定法解决"规则是什么"，上下文治理解决"规则如何到达 AI"。

一个常见误区是：规则写得越多、AI 每次读得越全，行为就越可控。实际相反——**上下文是稀缺资源**，规则的边际效力随总量递减。上下文治理的目标是：

> 用最少的常驻上下文，让 AI 在每个任务里恰好看到它需要的规则与事实。

## 1. 为什么上下文是稀缺资源

三种真实的损耗机制：

| 损耗 | 机制 | 后果 |
|---|---|---|
| 注意力稀释 | 规则越多，每条规则的遵守度越低 | 读了五份宪法的 agent，对每份的执行都打折扣 |
| 关键信息淹没 | 硬约束混在背景叙述里 | 真正的红线被当成普通建议 |
| 成本与延迟 | 每个会话都要付全量阅读成本 | 高频协作下，浪费按次数放大 |

所以上下文治理的第一原则不是"写清楚"，而是**"少而准"**：每多一条常驻规则，都在向已有规则征收注意力税。

## 2. 常驻与按需：规则的两种加载方式

判断一条规则该不该常驻（进全局 / 项目级 AGENTS、CLAUDE 之类每会话必读文件），门槛只有一个：

> **每次会话都要用到，且别处没有。**

两个条件缺一不可：

- "每次都要"不满足 → 沉到专用文档，按需加载。
- "别处没有"不满足 → 已有权威源（代码、lint 配置、架构文档）的内容不要复述，复述必然漂移。

| 内容 | 归宿 | 加载方式 |
|---|---|---|
| 硬红线（每次必守的禁令/约定） | 常驻规则文件 | 每会话自动 |
| 完整规则 + 历次违反档案 | 专用 conventions / lessons 文档 | 常驻文件留一行指针 |
| 架构蓝图、契约细节 | 工程宪法 / 契约文档 | 任务相关时按需读 |
| 一次性任务上下文 | 计划 / handoff 文档 | 当次任务 |

## 3. 薄宪法模式

常驻规则文件应该是**薄宪法**：只留要义 + 一行指针，全文和台账沉到专用文档。

```text
❌ 厚宪法：完整规则 + 示例 + 历次违反记录，全部堆在常驻文件里
✅ 薄宪法：一行要义 + 指向专用文档的指针
```

示例（常驻文件里的一行）：

```markdown
## 工程质量硬线（完整规则 + 违反档案见 specs/code_quality_lessons.md）
1. 纯逻辑必有单测 —— 详见该文档 §2
```

### 3.1 膨胀信号

出现以下情况，说明常驻文件该瘦身了：

- 单条规则超过 5 行（细节应该沉下去）。
- 出现"背景说明""历史原因"段落（这是文档内容，不是规则）。
- 新会话的 AI 开始违反文件前半部分的规则（后加的内容稀释了先有的）。
- 同一主题的规则散落多处（该合并成一条要义 + 一份专用文档）。

### 3.2 沉淀纪律

规则升级（见《技术债与经验回流》）产生的新规则，默认落**专用文档**，常驻文件只加指针行。把"沉淀到哪一层"作为规则升级提案的必填字段，防止一切新规则都涌向常驻文件。

## 4. 必读矩阵：按任务类型路由

项目文档多了之后，第二个膨胀点是"必读列表"：每份重要文档都被标成必读，每个会话都付全量阅读成本。

治理方式是把"必读列表"改成**必读矩阵**——按任务类型路由：

```markdown
| 文档 | 建页面 | 改基座 | 改埋点 | 对账迁移 |
|---|:-:|:-:|:-:|:-:|
| 架构蓝图 | ⭐ | ⭐ | | |
| 页面建设 playbook | ⭐ | | | ⭐ |
| 埋点契约 | | | ⭐ | |
| 源裁决规则 | ⭐ | | | ⭐ |
```

收益是双重的：省下的不只是 token，更是注意力——只读三份相关文档的 agent，对每份的遵守度都高于读七份的。

## 5. 子 agent 上下文契约

子 agent 是**无状态**的：看不到主对话历史，看不到之前其他 agent 的产出，也看不到用户和主 agent 达成的口头共识。由此推出一组硬契约：

### 5.1 输入：prompt 必须自包含

- 计划全文、变更文件列表、项目根路径、判断清单——凡是子 agent 需要的，主 agent 必须在 prompt 内**显式注入**或明确指路（"自行 Read 某文件"）。
- 不能写"按我们之前讨论的"——子 agent 没有"之前"。
- 对话里达成、未落盘的决策，主 agent 是唯一载体；要么写进 prompt，要么先落盘再指路。

### 5.2 输出：只返回数据

- 子 agent 的返回值是给主 agent 消费的数据，不是给人看的汇报——要求"findings 列表，无问题返回空列表，不要客套"。
- 返回格式在 prompt 里显式定义（字段、结构），否则每个 agent 自由发挥，主 agent 的汇总成本暴涨。

### 5.3 责任：主 agent 是上下文的搬运者

子 agent 空跑、比错文件、误解边界，多数根因是主 agent 注入的上下文不完整或有歧义。**子 agent 的输出质量上限，由主 agent 给的输入决定。**

## 6. Token 经济：机械事不派 agent

派一个子 agent 的成本 = 上下文注入 + 模型调用 + 结果消化。这笔账决定了分工：

- **机械 / 可 grep 的事主 agent 自己做**：跑几条命令看输出、grep 核实存在性——派 agent 纯属浪费，还引入转述失真。
- **判断的事才派 agent**，且按难度伸缩：琐碎 0 个、常规 2 个、高危才拆细加对抗复核。
- 模型档位随角色分层，见《下层执行》§5.4 子 agent 模型经济学。

## 7. 长对话的上下文治理

单个会话内，上下文同样会稀释：

- **重要结论及时落盘**：对话中期达成的关键决策，写进计划 / 决策文档，不依赖"对话还记得"。压缩、续作、交接时，落盘的才存在（handoff 机制见《长任务、Loop、Cron 与无人值守》）。
- **验证产物物证化**：跑过什么验证、覆盖了什么路径，写成一行"已验证 / 未覆盖"，而不是散落在过程输出里。

## 8. 上下文治理检查清单

为项目做治理设计时，检查：

- [ ] 常驻规则文件是否只含"每次都要 + 别处没有"的内容？
- [ ] 是否有专用 lessons / conventions 文档承接完整规则与台账？
- [ ] 常驻文件是否有膨胀信号（单条 >5 行 / 背景段落 / 散落重复）？
- [ ] 必读文档是否按任务类型路由，而不是全量 ⭐？
- [ ] 派子 agent 的 prompt 是否自包含（计划全文 / 文件列表 / 返回格式）？
- [ ] 是否规定了子 agent 只返回数据？
- [ ] 机械核查是否留在主 agent，没有浪费在子 agent 上？
- [ ] 对话中期的关键决策是否有落盘纪律？
