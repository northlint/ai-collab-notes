# 上层定法：架构不变量、业务 Spec 与真相源治理

## 核心定义

上层定法，是给 AI 协作建立“法源”和“裁决权”。

它不负责写死所有项目细节，而负责回答：

- 项目的稳定规则是什么？
- 哪些东西不能被任务随手改掉？
- 业务能力的当前态在哪里？
- 多份材料冲突时谁说了算？
- AI 不确定时应该停下，还是允许推断？
- 新经验应该沉淀到哪一层？

上层定法的关键是：

> 上层不管所有细节，但必须掌握裁决权。

## 1. 架构不变量：项目的宪法

### 1.1 什么是架构不变量

架构不变量是项目长期成立的工程规则。

它不是一次需求的计划，也不是历史记录，而是当前生效的约束。

它定义：

- 系统分几层
- 依赖方向如何
- 统一入口在哪里
- 数据和状态如何流动
- 错误、权限、日志、埋点、网络、UI 等基础能力由谁管理
- 哪些做法永远禁止

### 1.2 为什么 AI 需要架构不变量

AI 很擅长局部补全，但局部合理不代表全局正确。

没有架构不变量时，AI 会为了完成眼前任务：

- 在页面里直接请求接口
- 绕过统一错误处理
- 把服务端状态塞进 UI store
- 新造一个组件系统
- 复制一套已有逻辑
- 为了修一个 bug 改掉公共契约

这些改动单看都可能“能跑”，但会持续侵蚀项目结构。

### 1.3 不变量的典型类型

| 类型 | 作用 | 示例 |
|---|---|---|
| 分层不变量 | 规定模块依赖方向 | 页面只能依赖业务模型，不能绕过模型直接调接口 |
| 数据流不变量 | 规定数据归属 | 服务端数据归 Query / Repository，本地 UI 态归 Store |
| 入口不变量 | 规定统一封装 | 网络、桥接、权限、日志、金额格式化必须走统一入口 |
| 契约不变量 | 规定跨端或跨模块协议 | bridge handler 名、响应信封、错误码语义统一 |
| 设计不变量 | 规定视觉系统 | 业务代码不能写裸颜色，必须消费 token |
| 质量不变量 | 规定验证底线 | 纯逻辑必须测试，测试名和断言必须一致 |
| 安全不变量 | 规定高风险行为 | git push、删除、合并、发版必须人工确认或走审批 |

### 1.4 不变量的治理原则

架构不变量必须满足：

- **当前态**：读它就知道现在规则是什么。
- **稳定**：不为一次任务随便变化。
- **可引用**：计划和审查能明确引用它。
- **可检查**：能自动化就自动化，不能自动化就进入 review checklist。
- **不可自改**：AI 不得一边违反规则，一边把规则改成允许自己违反。

### 1.5 修改不变量的规则

不变量不是永远不能变，而是不能被“顺手变”。

如果确实需要修改，应单独走规则变更流程：

1. 说明现有不变量哪里不适用。
2. 说明新规则解决什么问题。
3. 说明影响哪些模块和历史实现。
4. 说明迁移方式和兼容策略。
5. 经过计划审查后再改。

## 2. 业务 Spec：用户能力的当前态

### 2.1 计划和 Spec 的区别

计划是变更流，Spec 是当前态。

| 文档 | 回答的问题 | 生命周期 |
|---|---|---|
| 计划 | 这次要改什么，怎么改，怎么验收 | 一次任务 |
| 业务 Spec | 当前用户能做什么，系统应如何表现 | 长期有效 |
| 评审记录 | 这次发现了什么问题，如何处理 | 历史证据 |
| 技术债 | 哪些已知问题暂缓，何时偿还 | 开放账本 |

如果项目只有计划，没有 Spec，AI 下次只能从历史计划里猜当前状态。

这会导致：

- 已完成的计划继续被当成待办
- 已废弃的方案继续被引用
- 多个计划互相冲突
- 当前行为只能靠读代码反推

### 2.2 业务 Spec 应该写什么

业务 Spec 管“用户能力”，不管底层架构实现。

它应该描述：

- 用户在什么状态下看到什么
- 可以执行哪些动作
- 动作成功后发生什么
- 失败、取消、空态、异常如何处理
- 权限、门控、状态机是什么
- 与外部接口或系统有什么契约
- 哪些行为是有意简化，哪些是待补技术债

适合用 WHEN / THEN 写：

```text
WHEN 用户处于审核中
THEN 首页展示审核等待态
AND 主按钮不可发起新申请
AND 倒计时结束后触发状态刷新
```

### 2.3 业务 Spec 的沉淀时机

一次任务完成后，需要判断结果是否应合入 Spec。

应合入：

- 用户可见能力变化
- 状态机变化
- 门控规则变化
- 接口契约变化
- 错误处理策略变化
- 跨页面流程变化

不一定合入：

- 一次性实现细节
- 临时调试方案
- 局部重构过程
- 已归档历史争论

## 3. 真相源治理：谁管什么

### 3.1 为什么需要真相源治理

AI 面对多份材料时，最危险的不是“不读”，而是“读了但读错用法”。

常见错误：

- 用视觉稿判断功能是否存在。
- 用接口文档覆盖现网真实请求。
- 用 mock 数据推断后端契约。
- 用旧代码否定新架构。
- 用某次对话里的临时决定覆盖稳定规则。
- 用项目 A 的经验直接套到项目 B。

所以必须规定：

> 每个真相源只在自己的维度内有权威。

### 3.2 常见真相源类型

| 源类型 | 管什么 | 不管什么 |
|---|---|---|
| 架构源 | 分层、依赖、技术边界、基础设施 | 不决定业务功能是否存在 |
| 业务源 | 用户行为、流程、状态机、校验 | 不决定视觉像素细节 |
| 视觉源 | 布局、尺寸、颜色、图标、交互表象 | 不决定功能存废 |
| 契约源 | 接口字段、状态码、数据形状 | 不替代运行事实 |
| 运行事实源 | 现网真实请求、真实响应、真实端上行为 | 不自动推翻架构规则 |
| 历史决策源 | 为什么这么做，曾经踩过什么坑 | 不压过最新明确决策 |
| 用户即时指令 | 本次任务目标、优先级、边界 | 不应静默改写长期规则 |

### 3.3 冲突裁决的步骤

当源冲突时，不要直接问“哪个更可信”，而要按步骤裁决：

1. 先判断冲突发生在哪个维度。
2. 找该维度的权威源。
3. 判断是否存在更高层规则。
4. 判断是否有实测事实。
5. 若仍不能裁决，标记阻塞并回问，不允许脑补。

示例：

- 视觉稿没画某按钮，但业务源显示功能存在：功能保留，视觉缺口登记。
- 接口文档写字段是 string，但真实响应是 number：契约按真实响应修正，文档差异登记。
- 旧代码直接请求接口，但架构不变量要求统一 API 层：按架构不变量重写，不复制旧坏模式。

### 3.4 真相源声明模板

每个复杂计划都应声明：

```md
## 真相源

- 架构约束：
- 业务行为：
- 视觉体验：
- 接口契约：
- 运行事实：
- 历史决策：

## 冲突裁决

- 结构/技术冲突：
- 产品行为冲突：
- 视觉冲突：
- 接口路径/方法冲突：
- 字段类型/可空冲突：
- 现网事实冲突：
```

## 4. 约束源与变更流分家

### 4.1 为什么必须分家

约束源和变更流生命周期不同。

| 类型 | 性质 | 目标 |
|---|---|---|
| 约束源 | 当前态、稳定、长期有效 | 让 AI 知道现在规则是什么 |
| 变更流 | 一次性、过程性、任务相关 | 让 AI 知道这次要改什么 |

如果混在一起：

- 架构规则会被需求流水账淹没。
- 历史计划会被误认为当前规则。
- AI 会在过期文档里找依据。
- 上层法会被下层任务反向污染。

### 4.2 什么进约束源

适合进入约束源：

- 架构分层
- 目录规范
- 依赖方向
- 技术选型
- 数据流规则
- 测试策略
- 安全边界
- 跨端契约
- 审查流程

### 4.3 什么进变更流

适合进入变更流：

- 本次需求目标
- 本次任务范围
- 本次风险分析
- 本次分批计划
- 本次验收清单
- 本次开放问题

### 4.4 灰色地带处理

有些任务既改业务，又改底层约束。

处理方式：

1. 以业务变更为入口。
2. 在计划里显式说明会影响哪些约束源。
3. 对约束源修改单独列风险和验收。
4. 审查时同时审业务和约束变更。
5. 完成后把稳定结果合入当前态文档。

### 4.5 规则层级：全局规则、项目规则、工程宪法、业务变更流

上层定法不是一个文件能解决的。

成熟项目至少有四类规则源：

| 层级 | 管什么 | 典型载体 | 稳定性 |
|---|---|---|---|
| 全局工作规则 | AI 如何协作、计划、审查、交付 | 全局 AGENTS / CLAUDE / skill | 高 |
| 项目工作规则 | 本项目目录、命名、命令、禁用模式 | 项目 AGENTS / CLAUDE | 中高 |
| 工程宪法 | 系统怎么搭、架构不变量、技术边界 | specs / architecture docs | 高 |
| 业务变更流 | 用户能做什么、本次迭代改什么 | openspec / proposal / tasks | 中低 |

这四层不能混成一坨。

全局规则解决“AI 和人怎么协作”。

项目规则解决“这个项目怎么跑”。

工程宪法解决“代码应该按什么结构存在”。

业务变更流解决“这次用户能力怎么变化”。

### 4.6 specs 与 openspec 的分工

一个可迁移的口诀：

> specs 管系统怎么搭，openspec 管用户能做什么。

`specs/` 适合承载：

- 分层架构
- 模块边界
- 命名规则
- 禁用 API
- 设计 token
- 路由规则
- 数据流约束
- 测试策略

`openspec/` 适合承载：

- 一次业务 change
- proposal
- design
- spec delta
- tasks
- archive

如果把业务迭代全部塞进工程宪法，宪法会变成流水账。

如果把长期架构铁律塞进一次次业务 change，当前有效规则会散落在历史归档里。

所以两者必须分家。

### 4.7 规则冲突裁决顺序

当多个规则源冲突时，必须显式裁决。

推荐顺序：

```text
用户本次明确指令
  > 当前任务已确认决策
  > 项目级 AGENTS / CLAUDE
  > 项目工程宪法 specs
  > 当前业务 spec / openspec
  > 全局 AGENTS / CLAUDE 默认工作流
  > 历史计划 / 旧文档 / 经验猜测
```

这不是说全局规则不重要。

全局规则定义默认协作方式；项目规则定义当前项目的具体边界；用户本次指令可以临时覆盖，但应在计划或审查中显式记录。

### 4.8 业务驱动的基建改动

现实中常见灰区是：

> 一个业务需求必须顺手升级工程基建。

例如：

- 新功能需要扩展网络层。
- 新页面需要新增导航模式。
- 新状态需要改变缓存策略。

处理方式：

1. 仍以业务 change 为入口。
2. 在 design 或 plan 中列出附带基建改动。
3. 对基建改动单独说明风险和验收。
4. change-review 同时审业务完成度和架构一致性。
5. 稳定后把基建规则合入工程宪法。

这样业务可以推动基建演进，但不能偷偷污染约束源。

## 5. 知识中枢：点名中枢，以上御下

### 5.1 中枢不是什么

中枢不是：

- 所有文档的堆放处
- 项目细节大全
- 历史记录全集
- 代码索引替代品
- 临时计划垃圾桶

### 5.2 中枢是什么

中枢是路由器和裁决索引。

它告诉 AI：

- 架构不变量在哪里
- 业务 Spec 在哪里
- 当前 active plan 在哪里
- 技术债在哪里
- review 历史在哪里
- 真相源如何裁决
- 新经验该回流到哪一层

### 5.3 中枢的最小结构

一个项目的中枢至少应包含：

```text
README / INDEX
  - 当前项目定位
  - 文档分层
  - 真相源表
  - active plan
  - durable spec
  - technical debt
  - review history
  - workflow
```

### 5.4 以上御下的实际含义

“以上御下”不是上层写死下层所有做法，而是：

- 上层定义裁决权。
- 下层按裁决权执行。
- 下层发现新事实，可以上报。
- 下层不能擅自推翻上层。
- 如果要推翻，必须走规则变更。

一句话：

> 上层定边界，下层做实现；上层定法源，下层找事实；上层定流程，下层跑任务。

## 6. 上层规则如何变成执行能力

上层规则如果只停留在文档里，最终仍然依赖 AI 自觉。

更稳的路径是：

```text
规则文本
  → checklist
  → skill / workflow
  → script / lint / CI
  → 执行门禁
```

例如：

- “计划必须审查”应落成 plan-review skill。
- “改动后必须审查”应落成 change-review skill。
- “禁止裸值”应落成 lint 或 grep。
- “注释引用必须真实存在”应落成提交前自检。
- “review 发现重复问题要升级规则”应落成 history 分析。

一句话：

> 文档定法，workflow 执法，门禁验法，history 修法。

## 7. 上层定法的检查清单

新项目或项目治理改造时，先问：

- 是否有架构不变量？
- 是否有业务 Spec？
- 是否区分全局规则、项目规则、工程宪法和业务变更流？
- 是否区分当前态和历史计划？
- 是否定义了真相源类型？
- 是否定义了冲突裁决？
- 是否定义了规则冲突裁决顺序？
- 是否有技术债账本？
- 是否有 review 历史？
- 是否有经验回流路径？
- 是否明确 AI 不能擅自修改哪些上层规则？
- 是否有一个中枢索引让 AI 快速定位这些材料？

如果这些问题答不上来，AI 参与越深，风险越高。
