# 下层执行：计划、批任务拆解与主从编排

## 核心定义

下层执行，是把上层规则转化为一次次可控的任务执行。

它关注的不是“AI 能不能做”，而是：

- 任务如何进入？
- 怎么判断范围？
- 如何拆批？
- 哪些先串行，哪些能并行？
- 子 agent 的边界在哪里？
- 主 agent 如何集成和验证？
- 做到哪里必须停下来审查？

下层执行的目标是：

> 把开放式任务降维成可计划、可验证、可回滚的小闭环。

## 1. 任务进入：先识别任务类型

不同任务进入系统后，走不同执行策略。

| 任务类型 | 特征 | 推荐处理 |
|---|---|---|
| 小修小补 | 范围明确、低风险 | 直接执行 + 简短验证 |
| 复杂功能 | 多模块、多状态、多依赖 | 先计划、再 plan-review、再执行 |
| 架构/契约变更 | 影响上层规则或跨模块协议 | 单独成批，必须审查 |
| 批量迁移 | 多页面、多端、多文件重复工作 | 建 manifest、分批、统一 recipe |
| 自动修 bug | 外部任务输入、需要诊断和人类确认 | 队列 + 诊断 + 人工卡点 |
| 长任务续作 | 会话可能中断或跨时间执行 | handoff 文档 + cron/watch/loop |
| 无人值守 | AI 自主跑一段时间 | 需要 oracle、门禁、审核队列 |

不要让所有任务走同一流程。

## 2. 计划：把任务变成可控变更单元

复杂任务必须先写计划。

计划不是为了形式，而是为了让 AI 明确：

- 做什么
- 不做什么
- 听谁的
- 怎么拆
- 怎么验收
- 什么时候停

### 2.1 计划必须包含的内容

```md
## 目标
本次要达成什么结果。

## 非目标
本次明确不做什么。

## 真相源
架构、业务、视觉、契约、运行事实分别来自哪里。

## 范围
会影响哪些模块、页面、接口、文档、测试。

## 执行步骤
按依赖顺序拆解。

## 风险
最可能错在哪里。

## 验收
怎么证明完成。

## 停止条件
什么情况下必须停下来审查或回问。
```

### 2.2 计划中的非目标很重要

AI 容易顺手扩张。

非目标的作用是告诉 AI：

- 看到相关问题也不要顺手修。
- 发现新需求先登记，不直接做。
- 不要把本批变成“顺便重构”。
- 不要把临时简化伪装成完成。

## 3. 批任务拆解

批任务的核心不是“平均分工”，而是按风险、依赖和验证方式拆。

### 3.1 不要按时间拆

错误拆法：

```text
今天做一半页面，明天做另一半。
```

更好的拆法：

```text
先做共享基建
再做彼此独立的页面
最后统一集成验证
```

AI 执行不适合用时间作为主要边界，因为时间不可控。

更好的边界是：

- 文件范围
- 模块边界
- 共享依赖
- 风险等级
- 验收方式
- 回滚粒度

### 3.2 常见拆解维度

| 维度 | 拆法 |
|---|---|
| 共享依赖 | 先做基建、组件、契约、mock，再扇出业务 |
| 风险等级 | 高风险写操作单独成批 |
| 验证方式 | 需要真机/真账号/截图的集中验证 |
| 业务边界 | 一页、一域、一能力、一状态机 |
| 变更性质 | 基建、功能、修复、测试、文档分开 |
| 回滚粒度 | 能独立回滚的一组作为一批 |

### 3.3 批次大小原则

一个批次应该满足：

- 一眼能审
- 可独立验证
- 可独立回滚
- 目标单一
- 不跨太多边界

如果一批同时包含：

- 新架构
- 新页面
- 新接口
- 新样式系统
- 新测试框架
- 文档迁移

那就过大了。

## 4. Phase 编排：Phase 0 / Phase 1 / Phase 2

批任务中最稳定的编排模式是三段式。

```text
Phase 0：共享基建串行先行
Phase 1：独立任务并行扇出
Phase 2：主 agent 集成验证
```

### 4.1 Phase 0：共享基建串行先行

Phase 0 做所有会被多个子任务依赖的东西。

包括：

- 目录结构
- Query / API / Repository 基座
- key 工厂
- mock / fixture
- i18n 空块
- 共享 UI 类或组件
- 公共类型
- 测试工具
- lint / grep / dep rules

原则：

> 共享件没就绪，不允许扇出并行。

否则子 agent 会各自补一套共享件，造成冲突和分叉。

### 4.2 Phase 1：独立任务并行扇出

Phase 1 可以并行，但必须限制边界。

典型规则：

- 一个 agent 只负责一个页面、一个模块或一个能力。
- 只能改自己的文件。
- 不得改共享件。
- 需要共享件改动时，停下报告主 agent。
- 每个 agent 做逻辑级自测。
- 不要求每个 agent 做最终像素/真机/全量验证。

这样并行才安全。

### 4.3 Phase 2：主 agent 集成验证

Phase 2 必须由主 agent 串行完成。

原因：

- 子 agent 看不到完整上下文。
- 子 agent 容易各自为政。
- 视觉、真机、真账号、console、端到端验证必须统一口径。
- 集成问题往往跨文件，不能分散处理。

Phase 2 做：

- 全量静态检查
- 构建
- grep 闸门
- 浏览器/真机验证
- 截图对账
- console 检查
- 文档同步
- change-review
- 修复集成问题

### 4.4 Phase 纪律

phase 不是时间盒，而是可审查、可回滚的变更单元。

推荐纪律：

- 单个 phase 控制文件数量，避免一批大到无法审查。
- 单端任务不跨端混改。
- 单类型任务不混做，例如基建、配置、测试、feature、文档分开。
- 不跨 phase 合并执行。
- 每个 phase 结束后做 change-review。
- 高风险 phase 结束后再 commit / tag / 归档审查记录。
- 终止条件不基于“时间到了”，而基于 scope 失控、返工率高、真相源不清或验证无法完成。

phase 的目标不是让 AI 忙满一段时间。

phase 的目标是让人类能看懂、能审完、能回滚。

### 4.5 Scope 失控的信号

出现这些情况，应停止当前 phase，重新拆解：

- 文件数量显著超过计划。
- 任务同时触碰 UI、接口、架构、测试、文档多类边界。
- 为了完成当前任务，不断新增共享抽象。
- 子 agent 需要改共享件才能继续。
- 验收方式从简单自测升级成真机、真账号、多状态矩阵。
- 改动开始依赖未确认业务规则。
- 发现旧文档和当前事实冲突。

停止不是失败。

停止是把失控变更重新变成可治理变更。

## 5. 子 agent 与主 agent 分工

### 5.1 子 agent 适合做什么

子 agent 适合做：

- 独立页面
- 独立模块
- 单一域 DTO 移植
- 单一审查维度
- 单一文档章节
- 重复但边界清晰的工作

### 5.2 子 agent 不适合做什么

子 agent 不适合做：

- 改共享基建
- 改全局架构
- 做最终集成判断
- 做跨任务裁决
- 处理用户临时决策
- 做高风险 git 操作

### 5.3 主 agent 的责任

主 agent 不是简单调度器，而是集成负责人。

必须负责：

- 定义 Phase 0 共享件
- 给子 agent 明确边界
- 检查子 agent 是否空跑
- 抽查关键文件
- 统一集成验证
- 汇总文档与技术债
- 发起 change-review
- 处理 must-fix

经验教训：

> 子 agent 偶发空跑或误解边界，主 agent 必须核对交付，不能只看报告。

### 5.4 子 agent 模型经济学

分完工，还要分**模型档位**。一刀切（全用大模型 = 烧钱；全用小模型 = 烧质量）都是错的。按角色三档：

| 档位 | 角色 | 模型 |
|---|---|---|
| 检索 / 广撒网 | 找调用方、扫同类实现、拉资料 | 小模型 |
| 常规判断 | 规范核对、清单式预检 | 小模型可胜任（有清单兜底） |
| 核心判断 / 对抗复核 | 正确性主判断、blocking finding 证伪 | **主力模型，不降级** |

判断标准一句话：

> 输出**直接决定 blocking verdict** 的 agent 不降级；输出还需主 agent 消化核实的可降级。

一个反直觉的隐性成本：给弱模型审查加的防误报补丁（"必须引用 file:line""主 agent 先自行核实再采纳"）本质是**主 agent 在为弱模型买单**——弱模型省下的钱，以主 agent 复核成本的形式还回来。总则：**省钱在检索，保质在核查**。

## 6. 逐页 / 逐模块 Recipe

对重复任务，应沉淀 recipe。

一个典型 recipe：

1. 读输入源：架构、业务、视觉、契约、现有实现。
2. 做范围裁定：哪些做，哪些不做，哪些登记债务。
3. 接数据：只经规定层，不绕过统一入口。
4. 渲染或实现：遵守样式、埋点、状态、错误处理规则。
5. 自测：本模块逻辑路径跑通。
6. 回填文档：状态、债务、决策、跳转关系。
7. 交给主 agent 集成验证。

Recipe 的价值是：

- 降低每次重新商量的成本
- 防止 AI 漏步骤
- 让新 agent 可以接手
- 让批任务可并行

迁移 / 重写场景下 Recipe 的输入源怎么定（多源分维度授权）、"一致"怎么证明（parity 对账），见 [`迁移与重写治理.md`](./迁移与重写治理.md)。

## 7. 状态分类与接线

批量还原或页面构建中，状态必须先分类。

一种通用分类：

| 类型 | 含义 | 处理方式 |
|---|---|---|
| 持久业务状态 | 页面长期展示的状态，如空态、审核中、失败 | 可进入状态切换器或测试矩阵 |
| 触发弹窗 | 点击按钮出现的瞬时态 | 通过真实按钮触发，不作为主状态 |
| 流程步骤 | 表单步骤、确认步骤、推进步骤 | 通过 Next / Confirm 到达 |
| 共享浮层 | 多页复用的 sheet/dialog/card | 抽共享件，统一接线 |

先分类，再实现。

否则容易：

- 把弹窗做成伪页面状态
- 把真实流程藏进调试切换器
- 重复实现共享浮层
- 漏掉状态回填

## 8. 验收设计

验收必须分层。

| 层级 | 谁做 | 验收内容 |
|---|---|---|
| 子任务自测 | 子 agent | 自己范围的逻辑、基本渲染、局部测试 |
| 集成验证 | 主 agent | 全量构建、跨模块、视觉、console、真机 |
| 对抗审查 | review agent / 主 agent | 范围、架构、正确性、验证充分性 |
| 人工卡点 | 用户/负责人 | 产品判断、风险接受、合并/发版 |

验收不应该只靠一个环节。

## 9. 停止条件

执行中遇到以下情况应停止或升级：

- 真相源冲突无法按规则裁决
- 需要修改架构不变量
- 发现新需求超出本批范围
- 需要高风险写操作
- 关键验证环境不可用
- 视觉源缺失且无法合理降级
- 契约不明且影响真实请求
- 子 agent 需要改共享件
- 自动化验证失败且原因不明

停止不是失败。

对 AI 来说，知道何时停比盲目继续更重要。

## 10. 下层执行检查清单

开始执行前确认：

- 本任务类型是什么？
- 是否需要先写计划？
- 是否声明了非目标？
- 是否声明了真相源？
- 是否有明确验收？
- 是否拆成可审的小批次？
- Phase 0 共享件是否先行？
- Phase 1 并行边界是否清楚？
- Phase 2 集成验证由谁负责？
- 什么情况下必须停？

执行完成后确认：

- 计划内目标是否完成？
- 是否有越界改动？
- 是否产生新技术债？
- 是否需要更新 Spec？
- 是否需要更新 workflow 或 checklist？
- 是否已经进入 change-review？
