# 工具链治理：给 AI 造的轨道自己也会腐化

## 核心定义

Workflow、skill、审查命令、钩子脚本——这些为 AI 协作搭的"轨道"，本身也是代码：会有死代码、会有契约断线、会和文档脱节、会被 agent 绕过。

工具链治理回答三个问题：

- 轨道自身按什么工程标准维护？
- 一条约束应该由**谁**执法——文档、审查，还是机器？
- 无人值守时，怎么保证 agent 绕不过轨道？

## 1. Skill 也是代码

### 1.1 适用同一套工程纪律

| 代码的纪律 | 在 skill 上的形态 |
|---|---|
| 单一职责 | 一个 skill 管一段流程；诊断的不动手术，审查的不顺手修 |
| 死代码清理 | 永远走"不存在则跳过"分支的步骤，按治理债处理（见《技术债与经验回流》§4.1） |
| 变更要 review | 改 skill 机制 = 改流程契约，值得和改代码一样对待 |
| 文档同步 | skill 机制变更后，引用它的方法论文档欠一次同步——改了没同步 = 没改完 |

### 1.2 跨工具读写契约：两端一起设计

工具链最隐蔽的断线：工具 B 设计为读取工具 A 落盘的文件，而 A 的定义里从没有写盘步骤。B 每次都走"文件不存在 → 跳过"，机制从未生效，也从不报错（完整案例见《案例库》#23）。

规则：

- 读端定义时，写端必须同时写明：**由谁、在哪一步、写什么格式**。
- 读端注明写端出处，让断线可审计。
- 连续 N 次全走跳过分支的机制，按治理债清理。

## 2. 执法层级：约束由谁兜底

同一条约束，可以由五个层级执法。层级越高，agent 绕过的可能性越低，建设成本也越高：

| 层级 | 执法者 | agent 能否绕过 | 典型形态 |
|---|---|---|---|
| L1 模型自律 | agent 自己 | 随时（无意漂移即绕过） | 口头约定、对话提醒 |
| L2 文档规则 | 常驻规则文件 | 能（注意力稀释、规则冲突时） | AGENTS / CLAUDE 条目 |
| L3 review 执法 | 审查环节 | 较难（但审查本身可能被跳过） | plan / change review checklist |
| L4 机器闸门 | 脚本 / lint / CI | 很难（除非改闸门本身） | lint 规则、grep 闸门、测试 |
| L5 harness 硬拦 | 运行环境钩子 | **不能**（在 agent 够不着的层级） | pre-commit 拦截、权限钩子 |

两条推论：

- **规则升级的终点是升层级**，不是把文档写得更严厉。同类问题反复出现时，问的不是"要不要再强调一遍"，而是"能不能升到 L4/L5"。
- **越接近无人值守，要求的执法层级越高。** 有人盯着时 L2/L3 够用；无人时，关键约束必须在 L4 以上——L1/L2 靠的是 agent 自觉，而无人值守的前提恰恰是不依赖自觉。

## 3. Verdict 物证化

无人值守流程的自动推进，凭据必须是**物证**，不是声称：

- 审查通过与否，只认落盘文件里的 verdict 行；agent 口头说"review 过了"无效——声称无法被事后证伪，物证可以。
- 硬闸门装在 L5：无人值守标记存在时，提交动作被运行环境钩子拦截，验证落盘 verdict == APPROVE 才放行。agent 哪怕漂移了，也提交不出去。
- 标记文件圈定生效范围：进入无人值守时创建、退出时删除——闸门不干扰有人值守时的正常裁量（如显式跳过 review）。
- 每次自动通过留一行审计记录，信任但可抽查。

同理，"验证做过了"也要物证化：检查点报告"已验证 / 未覆盖"一行（见《长任务、Loop、Cron 与无人值守》§5.6），而不是散落在过程叙述里。

## 4. 假绿防御是闸门的自证义务

L4 闸门自己也会腐化成假绿：扫描路径失效、规则空跑、测试数为 0，却仍然报 SUCCESS。所以闸门必须**自证执行**：

- 输出检查对象数量（扫描文件数 / 测试数 / 断言数），计数为 0 按失败处理。
- 定期故意造一个已知违例，确认闸门真的能抓到。
- 完整清单见《长任务、Loop、Cron 与无人值守》§5.7 假绿防御。

## 5. 方法论、模板与工具的三态关系

同一套机制往往存在三种形态，必须分清谁是权威：

| 形态 | 角色 | 权威范围 |
|---|---|---|
| 方法论文档（本知识库这类） | 讲**原理与设计理由**，自包含、可迁移 | why |
| 模板 | 可复制的骨架 | 起点 |
| skill / 脚本（落地实现） | **运行时准绳** | how 的最终裁决 |

规则：

- 机制落地成 skill 后，运行时以 skill 为准；方法论文档记录 why，与 skill 保持**机制级一致**（verdict 枚举、状态语义、落盘契约、闭环走向），措辞级差异不算脱节。
- skill 机制变更 → 方法论文档欠一次同步。同步纪律本身也要遵守执法层级：只写"要记得同步"是 L2，配一个哈希基线比对脚本才是 L4。

## 6. 工具链治理检查清单

- [ ] 每个 skill 职责单一，没有顺手越界的步骤？
- [ ] 跨工具的读写契约两端齐全，读端注明写端出处？
- [ ] 有没有连续空转的流程环节（治理债候选）？
- [ ] 每条关键约束标注了执法层级？无人值守路径上的约束 ≥ L4？
- [ ] 自动推进的凭据是落盘物证，不是口头声称？
- [ ] 机器闸门有自证执行（计数 > 0）？
- [ ] 方法论 / 模板 / skill 三态的权威关系已声明？同步纪律有机械触发物？
