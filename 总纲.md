# 总纲：AI 协作工程治理系统

## 一句话定义

AI 协作工程治理系统，是一套让 AI 在长期软件工程中可控、可审查、可验证、可演进的工程方法。

它的核心是：

> 上层定法，下层执行，审查纠偏，经验回流。

展开后是两条主线：

- **治理线**：定义规则、约束、真相源、审查、技术债和经验沉淀。
- **执行线**：定义任务如何进入、拆解、调度、续作、无人值守和交付。

这套体系的目标，不是让 AI “更自由地干活”，而是让 AI 在正确边界内高效干活。

## 为什么不是“写代码技巧”

AI 写代码的问题，很多时候不是能力不足，而是工程协作方式本身没有治理结构。

没有治理结构时，AI 容易出现五类失效。

| 失效 | 表现 | 后果 |
|---|---|---|
| 越界 | 顺手重构、顺手改架构、顺手引入新依赖 | 改动面失控，难审难回滚 |
| 漂移 | 文档说 A，代码做 B；下次会话忘记本次约定 | 项目长期不可维护 |
| 脑补 | 源冲突时靠猜，缺材料时自己编 | 产生看似合理但实际错误的实现 |
| 假完成 | 自称做完，测试没断言，失败路径没处理 | 信任被消耗，人工审查成本暴涨 |
| 遗忘 | 同类错误反复出现 | 人一直在重复纠错，系统没有变强 |

所以，真正的问题不是“怎么 prompt AI 写好一点”，而是：

> 如何把项目本身建设成 AI 能正确操作的工程系统。

## 从代码仓库到治理系统

传统代码仓库主要包含：

- 代码
- 测试
- 配置
- 少量 README

AI 深度参与后，仓库还需要包含：

- 架构不变量
- 业务 Spec
- 真相源裁决规则
- Workflow
- Plan Review / Change Review
- 技术债账本
- 经验回流记录
- 自动化检查
- 长任务与无人值守执行协议
- 人工卡点与权限边界

这些不是“额外文档负担”，而是让 AI 可治理的基础设施。

## 四段闭环

```text
上层定法
  定义原则、架构不变量、业务 Spec、真相源、裁决权
      ↓
下层执行
  把任务拆成可控批次，按计划、边界和验收执行
      ↓
审查纠偏
  用对抗性审查发现方向错、实现错、验证不足
      ↓
经验回流
  把问题写入 review / 技术债 / recurring pattern / 规则
      ↓
回到上层定法
  让下一轮 AI 的自由度更窄，错误更早暴露
```

这四段不是一次性流程，而是一个飞轮。

真正成熟的系统，应该让问题越来越早暴露：

- 第一次靠人审查发现。
- 第二次靠 checklist 提醒。
- 第三次靠规则阻断。
- 稳定后靠自动化门禁拦住。

## 治理线与执行线

### 治理线

治理线关心的是“法”。

它回答：

- 项目的当前规则是什么？
- 谁是架构权威？
- 谁是业务权威？
- 谁是契约权威？
- 计划和 Spec 有什么区别？
- 技术债怎么登记？
- 审查怎么阻断错误？
- 经验如何升级成规则？

治理线的关键词：

- 架构不变量
- 业务 Spec
- 真相源治理
- 审查
- 技术债
- 经验回流
- 自动化门禁

### 执行线

执行线关心的是“术”。

它回答：

- 一个任务怎么进入系统？
- 长任务怎么不中断？
- 批任务怎么拆？
- 子 agent 怎么并行？
- cron 和 watch 怎么选？
- 无人值守什么时候成立？
- 人类在哪些点必须卡住？
- 执行器失败后怎么恢复？

执行线的关键词：

- 批任务
- Phase 0 / Phase 1 / Phase 2
- loop
- cron
- watch
- worker
- 队列
- 人工卡点
- oracle
- park-don't-guess

## 适用边界

这套体系不是给所有项目都上满。

| 场景 | 是否适用 | 说明 |
|---|---|---|
| 一次性 Demo | 不适合全套 | 目标是快速验证，治理成本可能超过收益 |
| PoC / 原型 | 部分适用 | 需要真相源和边界，但不一定需要完整审查/债务体系 |
| 长期内部工具 | 逐步适用 | 先上计划、审查、技术债，再补不变量和自动化 |
| 生产项目 | 强适用 | 需要完整治理闭环 |
| 跨端 / 多仓协作 | 强适用 | 真相源和契约裁决尤其关键 |
| 自动修 bug / 无人值守 | 强适用 | 必须有队列、卡点、自验证、权限边界 |
| 大规模迁移 | 强适用 | 必须有冻结真相源、状态矩阵、批任务拆解和 oracle |

判断标准不是项目大小，而是：

> 这份代码和知识是否会被长期依赖。

如果会，就需要治理。

## 体系里的几个根本判断

### 1. 不靠 AI 自觉，靠系统约束

AI 不稳定，不是因为它“不认真”，而是因为它没有天然工程责任边界。

所以不要指望它自动：

- 控制范围
- 找权威源
- 不脑补
- 补文档
- 记住历史教训
- 主动停下来问人

这些都必须系统化。

### 2. 上层不管细节，但必须管裁决权

“以上御下”不是让上层写死一切细节。

上层不应该写：

- 某项目具体接口路径
- 某页面具体组件
- 某仓库具体命令

上层应该规定：

- 接口路径从哪类源来
- 页面行为由谁裁决
- 架构不变量在哪里
- 冲突时怎么处理
- 不确定时是否允许猜

下层负责执行，上层负责裁决。

### 3. 计划是变更流，Spec 是当前态

计划回答：

> 这一次要改什么？

Spec 回答：

> 现在系统是什么？

AI 最容易从历史计划里猜当前状态。长期项目必须把完成后的稳定结果沉淀进当前态 Spec，而不是让计划文档无限堆积。

### 4. 技术债不是借口，而是显性账本

可以暂缓，但不能静默省略。

任何简化、占位、未验证、未接入、未覆盖的路径，如果本轮不做，就要登记：

- 现象
- 风险
- 触发条件
- 暂缓理由
- 偿还方式

### 5. 无人值守不是“没人管”

无人值守的本质不是让 AI 完全自由，而是让它在强约束内自治。

无人值守成立的前提：

- 真相源冻结
- 有可运行 oracle
- 有状态矩阵或 fixture
- 有机械门禁
- 有失败恢复
- 有人工审核队列
- 不确定时能 park，而不是 guess

没有这些条件，所谓无人值守只是无人看管的风险累积。

## 成熟度模型

可以用五级判断一个项目的 AI 协作成熟度。

| 等级 | 状态 | 特征 |
|---|---|---|
| L0 临时使用 | AI 只按对话写代码 | 无长期规则，靠人工逐行看 |
| L1 有计划 | 复杂任务先写计划 | 有目标、范围、验收，但未形成当前态 |
| L2 有审查 | Plan / Change 都审 | 能阻断明显错误 |
| L3 有治理 | 架构不变量、Spec、技术债齐全 | AI 有明确上层法 |
| L4 有回流 | recurring 能升级规则或门禁 | 系统越用越紧 |
| L5 可自治 | loop / worker / oracle / 人工队列齐全 | 可承担长任务和部分无人值守作业 |

很多项目不需要直接到 L5。

但只要开始让 AI 承担长期生产任务，就至少应该达到 L3。

## 本知识库如何使用

建议按问题类型进入。

如果你在设计项目治理：

- 读 `上层定法.md`
- 重点建立架构不变量、业务 Spec、真相源治理

如果你在安排一次复杂任务：

- 读 `下层执行.md`
- 重点使用计划、批任务拆解、Phase 编排

如果你在设计自动化或长任务：

- 读 `长任务_loop_cron_无人值守.md`
- 重点判断 watch / worker / cron / loop 的适用边界

如果你在做质量控制：

- 读 `审查纠偏.md`
- 重点建立 Plan Review / Change Review

如果你在沉淀经验：

- 读 `技术债与经验回流.md`
- 重点把问题从 review 推到 debt / recurring / rule

如果你需要给别人解释：

- 读 `案例库.md`

如果你要直接执行：

- 读 `模板库.md`
