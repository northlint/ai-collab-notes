# 案例库

## 使用方式

案例库不是故事集，而是训练判断力的材料。

每个案例都按三段写：

- 反模式：AI 或团队容易怎么做错。
- 正解：应该如何纠偏。
- 抽象规则：沉淀成可复用的治理原则。

案例可以被用于：

- onboarding
- review checklist
- spec 模板示例
- 自动化工作流设计
- 子 agent 任务说明
- 技术债复盘

## 1. 把视觉源误当业务源

### 反模式

AI 根据设计稿里的文字、按钮顺序、状态标签，直接推断业务规则。

例如：

- 设计稿里有“审核中”，AI 就假设接口状态字段一定叫 `pending`。
- Figma 页面没有展示失败态，AI 就认为失败态不存在。
- 视觉稿里列表只有三条数据，AI 就把分页逻辑省掉。

### 正解

视觉源只回答“长什么样”。

业务规则必须从业务 spec、生产行为、接口事实、已验证逻辑中确定。

如果视觉源和业务源冲突：

- 先标记冲突
- 明确真相源优先级
- 必要时请求人工裁决
- 裁决后回写 spec

### 抽象规则

> 设计稿是视觉真相源，不是业务真相源。

## 2. 接口文档与生产事实冲突

### 反模式

AI 只读取旧接口文档，按文档字段实现。

但生产接口实际已经改字段，旧 mock 和旧测试又刚好支持旧字段，导致测试通过但线上失败。

### 正解

接口类任务必须区分：

- 历史文档
- 当前 openspec
- 真实请求样例
- 生产观测行为
- 现有代码调用事实

当这些来源冲突时，不能直接选择最容易读取的文档。

应按项目定义的真相源优先级裁决，并把冲突登记为文档债。

### 抽象规则

> 接口实现以当前事实为准，旧文档只能作为线索。

## 3. AI 声称测试通过，但没有断言

### 反模式

AI 增加测试文件，运行命令通过，然后声明完成。

审查发现：

- 测试只渲染组件
- 没有断言关键行为
- 没有覆盖异常路径
- mock 数据和真实风险无关

### 正解

测试必须证明任务目标。

至少要问：

- 测试断言了什么？
- 如果实现是错的，测试会失败吗？
- 是否覆盖了本次 bug 的复现路径？
- 是否覆盖边界状态？

### 抽象规则

> 测试通过不等于验证完成；没有断言的测试接近摆设。

## 4. 异步失败被静默吞掉

### 反模式

长任务里某一步 API 调用失败，catch 之后只打印日志。

主流程继续执行，最后输出“全部完成”。

### 正解

先定义错误等级：

- 阻断错误：停止流程，写入失败状态，通知人类。
- 可恢复错误：重试，超过上限后降级。
- 非阻断错误：记录 warning，不影响主流程。

自动流程结束前必须检查失败列表。

### 抽象规则

> 无人值守流程里，静默失败就是伪完成。

## 5. 技术债没有登记

### 反模式

AI 发现旧文档过期、某状态缺测试、某 worker 无复位，但因为本次任务还能完成，就只在最终回答里提一句。

下一次任务换了上下文，问题再次发生。

### 正解

凡是满足以下条件的问题，应登记技术债：

- 当前不修
- 后续会影响判断
- 有复发风险
- 有明确影响范围

登记时写清楚：

- 现象
- 风险
- 暂缓原因
- 偿还方式
- 验收标准

### 抽象规则

> 不登记的债务，会变成下一次 AI 的盲区。

## 6. 子 agent 空跑

### 反模式

主 agent 分配审查任务给子 agent。

子 agent 输出“未发现问题”，但报告里没有文件引用、没有命令证据、没有检查范围。

### 正解

子 agent 的产出必须可审计。

主 agent 应要求：

- 明确读取了哪些文件
- 明确审查维度
- 明确没有覆盖哪些范围
- 发现项带路径和证据

关键任务中，主 agent 必须抽查。

### 抽象规则

> 子 agent 能分担工作，不能分担最终责任。

## 7. Cron 续作导致重复工作

### 反模式

凌晨 cron 启动新会话，新会话不知道前一阶段已经完成，于是重复执行批任务。

结果：

- 覆盖前一次改动
- 重复提交
- 状态混乱
- 人类难以判断哪个结果可信

### 正解

cron 续作必须有 handoff。

handoff 要说明：

- 已完成什么
- 未完成什么
- 当前停在哪个检查点
- 哪些状态是正常停留
- 哪些动作不能重复
- 如何判断是否继续

cron 任务本身必须幂等。

### 抽象规则

> 没有 handoff 的 cron，是盲开新任务。

## 8. Watch 与 Worker 混用

### 反模式

本来只需要监听群消息，却让后台 worker 直接修代码、commit、通知。

由于 worker 无上下文，误把临时讨论当成正式任务。

### 正解

区分 watch 和 worker：

- watch 只观察事件，发现后退出，唤醒当前会话。
- worker 处理完整任务链，必须有锁、状态机、人工卡点。

当前窗口有人值守时，优先 watch。

真正无人值守且流程成熟时，才使用 worker。

### 抽象规则

> 能用 watch 保留上下文时，不急着上 worker。

## 9. 无头 worker 被权限和上下文卡住

### 反模式

为了自动化，把完整修复链交给无头 agent。

结果无头 agent：

- 没有当前会话上下文
- 工具权限不一致
- 等人工输入时卡死
- 错误处理不完整

### 正解

先采用 current-session 模式：

- 后台进程只监听
- 事件到达后退出
- 当前主会话继续处理

等工作流稳定后，再把明确、低风险、可状态化的部分下沉到 worker。

### 抽象规则

> 自动化成熟度不足时，先自动唤醒，不要自动决策。

## 10. 无人值守迁移没有 oracle

### 反模式

AI 批量迁移页面，运行构建通过后就认为完成。

但迁移目标其实包括：

- 视觉一致
- 业务状态一致
- 接口字段一致
- 真机交互一致
- 异常路径一致

构建通过无法证明这些。

### 正解

无人值守迁移必须提前定义 oracle。

oracle 可以包括：

- 源页面实时对照
- 状态矩阵
- schema 校验
- screenshot diff
- 真机操作脚本
- 控制台错误检查
- 网络请求断言

没有 oracle 的任务，不能承诺无人值守完成。

### 抽象规则

> 长任务能不能无人值守，取决于有没有可自动判断的完成标准。

## 11. Mock 被当成假数据

### 反模式

AI 把 mock 仅当成本地调试数据，随意改成能让页面好看的样子。

结果 mock 不再覆盖真实业务状态。

### 正解

在迁移、重构、状态机任务中，mock 应作为状态台账。

它应该覆盖：

- 成功
- 空态
- 加载
- 错误
- 边界值
- 权限差异
- 多状态组合

mock 改动必须说明对应业务状态。

### 抽象规则

> 高质量 mock 是状态 ledger，不是假数据。

## 12. “看起来正常”不等于设计还原

### 反模式

AI 打开页面，看一眼没有明显错位，就说 UI 完成。

但实际存在：

- 字体大小不一致
- 间距偏差
- 状态缺失
- 移动端换行错误
- 滚动区域不一致
- 图标资源错误

### 正解

设计还原需要明确验收方式。

常见方式：

- 与设计稿或源页面并排对照
- screenshot diff
- 多视口检查
- 关键组件状态截图
- 控制台错误检查
- 真机验证

### 抽象规则

> UI “能渲染”不是“还原完成”。

## 13. 缺资源后自由发挥

### 反模式

AI 找不到图标或图片，直接用占位图、渐变、纯色块替代，并把页面标记完成。

### 正解

资源缺失时应按顺序处理：

1. 搜索当前项目资源目录。
2. 搜索上游项目或旧版本。
3. 检查设计稿导出资源。
4. 询问或登记资产债。
5. 使用占位时明确标记不可最终验收。

### 抽象规则

> 缺资产不是自由创作授权，先找源，再占位。

## 14. 合约变更没有独立验证

### 反模式

AI 修改 schema、接口调用、UI 字段展示，但没有独立验证变更来源。

最后代码、mock、文档一起错。

### 正解

合约变更至少需要两类证据：

- 来源证据：字段来自哪里。
- 行为证据：实现如何被验证。

例如：

- openspec 示例
- 实际 API 响应
- 旧代码调用链
- 后端确认
- schema 测试

### 抽象规则

> 合约变更不能自证，必须有独立来源。

## 15. 批任务没有 Phase 0

### 反模式

多个页面、多个模块、多个 agent 同时开工。

每个 agent 自己建 mock、自己定义类型、自己处理缓存。

最后集成时发现：

- 类型重复
- 缓存失效策略冲突
- 目录结构不一致
- 组件边界不一致

### 正解

批任务要先做 Phase 0：

- 公共类型
- 共享 mock
- 数据契约
- 基础组件
- cache/invalidate 约定
- 验收标准
- 分工边界

然后再进入 Phase 1 并行实现。

### 抽象规则

> 并行之前先统一地基。

## 16. 人工卡点被省略

### 反模式

自动修复流程直接从诊断进入修改、commit、通知。

但诊断阶段存在业务判断，需要人类确认。

### 正解

把不可自动裁决的步骤设为人工卡点。

常见卡点：

- 需求歧义
- 业务规则冲突
- 删除数据
- 改公开接口
- push / merge / release
- 大范围重构

卡点消息要明确：

- 当前判断
- 需要人类选择什么
- 默认不做什么
- 超时如何处理

### 抽象规则

> 自动化不能越过业务裁决权。

## 17. Park-don't-guess

### 反模式

AI 缺少关键来源，但为了继续推进，猜一个最可能的方案。

猜测之后继续写代码，导致后续大量改动建立在不确定基础上。

### 正解

遇到关键不确定，应停在可恢复位置。

动作包括：

- 记录已知事实
- 记录缺失信息
- 说明不能推进的原因
- 给出下一步需要的输入
- 不做不可逆改动

### 抽象规则

> 缺关键真相源时，停住比猜对更工程化。

## 18. Commit 前没有核对实际 diff

### 反模式

AI 完成修改后直接 commit。

但格式化、生成文件、自动修复脚本引入额外改动，AI 没有检查。

### 正解

commit 前至少检查：

- `git status`
- 实际 diff
- 是否包含无关文件
- 是否有生成物或缓存
- 是否遗漏文档
- 是否通过必要验证

高风险任务还要做 change-review。

### 抽象规则

> commit 是交付动作，不是保存动作。

## 19. Review 报告没有 verdict

### 反模式

审查报告写了很多建议，但没有明确能不能继续。

执行 agent 不知道该修、该停、还是可以交付。

### 正解

review 必须给 verdict：

- `APPROVE`
- `APPROVE WITH CHANGES`
- `RETHINK`
- `REQUEST CHANGES`

并把问题分为：

- Blocking
- Important
- Minor

### 抽象规则

> 没有 verdict 的审查，不能驱动流程。

## 20. 经验只写在最终回复里

### 反模式

AI 在最终回答中总结“以后要注意”，但没有写入任何持久位置。

下一次任务完全无法继承。

### 正解

有复用价值的经验应回流到：

- 知识库
- AGENTS.md
- 模板库
- review checklist
- 技术债台账
- 自动化脚本

### 抽象规则

> 没有落盘的经验，不是系统能力。

## 21. 治理机制也会死亡

### 反模式

为"计划审查的 must-fix 被执行期遗忘"设计跨阶段账本：plan review 把问题落盘，change review 逐项核对关闭。机制精巧完整，运行后却发现账本永远为空——因为正确的流程本来就是执行前把 must-fix 直接修进计划、re-review 到通过。机制成了每次审查都要空跑一遍的死代码。

### 正解

废弃账本，源头修复：must-fix → 直接修订计划文档 → 增量 re-review → APPROVE 后才执行。同时把"账本之死"记录下来，防止后人重建。治理债的定义与判断标准见 [`技术债与经验回流.md`](./技术债与经验回流.md) §4.1。

### 抽象规则

> 治理机制防的问题若能在更源头零成本消掉，机制本身就是治理债。

## 22. 口头声称的 review 通过

### 反模式

无人值守 loop 中，agent 声称"review 已通过"，随即 commit 推进。实际 review 根本没跑，或跑了但 verdict 不是通过——声称无法被事后证伪，错误产物持续进入主干。

### 正解

verdict 物证化 + 分层执法：

- 自动推进的**唯一凭据**是审查落盘文件里的 verdict 行，口头声称无效。
- 在 agent 够不着的层级（运行环境的 pre-commit 拦截）加硬闸门：无人值守标记存在时，commit 必须验证落盘 verdict == APPROVE，否则直接阻断。
- 进入无人值守建标记文件、退出删除，让闸门只在无人时生效，不干扰有人值守的正常跳过。

### 抽象规则

> 口头声称不是物证；越接近无人值守，约束的执法层级必须越高。

## 23. 跨工具的读写契约断线

### 反模式

审查工具 B 设计为"读取工具 A 落盘的文件做闭环验证"，但工具 A 的定义里从头到尾没有写盘步骤。B 每次运行都命中"文件不存在 → 跳过"，这条精心设计的闭环**从未真正生效过**，且没有任何报错暴露它。

### 正解

- 跨工具的读写契约必须**两端一起设计**：读端定义时，写端必须同时写明"由谁、在哪一步、写什么格式"。
- 在读端注明写端出处（"此文件由 X 的第 N 步产生"），让断线可被审计。
- 定期对"条件跳过"分支做审计：连续 N 次全走跳过分支的机制，按治理债处理。

### 抽象规则

> 跨工具的闭环，先证明写端存在，再设计读端。
