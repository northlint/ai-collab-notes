# 模板库

## 使用方式

模板库用于把治理思想落到具体产物。

使用原则：

- 模板不是填空游戏，必须结合任务裁剪。
- 模板里的“不适用”也要说明原因。
- 涉及高风险任务时，宁可多保留检查项。
- 模板可以复制到项目 `plan/`、review 记录、handoff 文档或 AGENTS.md。

## 7. Phase 0 / 1 / 2 Playbook

```markdown
# Phase Playbook

## Phase 0：定地基
目标：让并行前的共同约束清楚。

输出：
- 真相源列表
- 架构不变量
- 数据契约
- 状态矩阵
- 公共组件 / 工具
- 验收标准
- 子任务边界

通过标准：
- 后续子任务不需要各自发明规则。

## Phase 1：分批执行
目标：在清楚边界内推进实现。

要求：
- 每个子任务有输入、输出、验收。
- 子 agent 不能修改共享契约，除非回到主 agent。
- 每批完成后记录状态。

通过标准：
- 子任务可以独立验证。

## Phase 2：集成验收
目标：证明整体一致。

要求：
- 合并后验证共享契约。
- 检查跨模块状态流。
- 检查视觉、接口、异常、缓存。
- 做 change-review。

通过标准：
- 不是每块都完成，而是整体可交付。
```

## 14. Git 写模式模板

```markdown
# Git Write Mode

## 当前模式
Mode A / Mode B / Mode C

## Mode A：只改不 commit
适合：
- 探索
- 用户要先看 diff
- 风险较高

限制：
- 不 commit
- 不 push

## Mode B：本地 commit，不 push
适合：
- 自动修复闭环
- 需要稳定保存
- 仍需人工合并

限制：
- commit 前必须核对 diff
- 不 push
- 不 merge

## Mode C：push / PR / merge
适合：
- 用户明确授权
- CI 和 review 规则完整

限制：
- 必须有人工授权
- 必须通过必要验证
```

## 23. 检查点协议模板

```markdown
# Checkpoint Protocol: {项目 / 工作流名称}

## 触发点
- 产出计划后（必停）
- 完成代码 / 内容改动后（必停）

## 选项（每次显式提供，不自动推进）
1. review（推荐）
2. 跳过，直接下一步
3. 讨论

## verdict 流转
- APPROVE → 等确认后下一步
- 有 must-fix → 修复 → 增量 re-review（只核对上轮问题）
- 根本性问题（RETHINK / REQUEST CHANGES）→ 修完必须重新 review，无跳过

## 无人值守分支
- 唯一自动分支：落盘 verdict == APPROVE → 自动继续
- 其余任何 verdict 一律停下等人（无"问题不大可继续"）
- verdict 只认落盘物证；自动通过留一行审计记录
- 进入前创建无人值守标记，退出删除（硬闸门以此判定生效范围）
```
